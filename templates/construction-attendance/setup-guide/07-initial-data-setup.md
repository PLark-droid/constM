# 07. 初期データ投入手順

> **所要時間**: 約30〜45分
> **前提条件**: テーブル構築（02）、ビュー設定（03）、フォーム作成（04）、自動化ルール設定（05）、ダッシュボード作成（06）が完了していること

## 概要

テーブル構築が完了した後、システムを稼働させるためにはマスタデータの投入が必要です。本ドキュメントでは、以下の3つのマスタテーブルに初期データを投入する手順を説明します。

| # | テーブル | サンプルCSV | サンプルレコード数 |
|---|---------|-----------|-----------------|
| T1 | 案件マスタ | `master-projects.csv` | 3件 |
| T2 | 作業員マスタ | `master-workers.csv` | 10件 |
| T3 | 作業内容マスタ | `master-work-types.csv` | 20件 |

投入方法は2つあります。

- **方法1: CSVインポート（推奨）** -- データ件数が多い場合や、既存の台帳データがある場合に最適
- **方法2: 手動入力** -- 少量のデータを直接入力する場合に利用

> **重要**: マスタデータの投入順序は **案件マスタ → 作業内容マスタ → 作業員マスタ** の順に行ってください。作業員マスタの「所属案件」フィールドは案件マスタへのリンクフィールドのため、先に案件マスタのデータが必要です。

---

## 事前準備: サンプルデータCSVのダウンロード

1. テンプレートリポジトリの `sample-data/` フォルダから以下のCSVファイルをダウンロードする

   | ファイル名 | 内容 | レコード数 |
   |-----------|------|-----------|
   | `master-projects.csv` | 案件マスタのサンプルデータ | 3件 |
   | `master-workers.csv` | 作業員マスタのサンプルデータ | 10件 |
   | `master-work-types.csv` | 作業内容マスタのサンプルデータ | 20件 |

2. 自社データに編集する場合は、ExcelまたはGoogleスプレッドシートでファイルを開いて編集する

3. 編集後は必ず **UTF-8エンコーディング** のCSVとして保存する

> **注意**: ExcelでCSVを開いて保存すると、エンコーディングがShift_JISに変わる場合があります。文字化けを防ぐため、「名前を付けて保存」から **CSV UTF-8（コンマ区切り）** を選択してください。

## 方法1: CSVインポート（推奨）

### Step 1: 案件マスタ（T1）のインポート

#### 1-1. サンプルデータの内容確認

`master-projects.csv` には以下の3案件が定義されています。

| 案件名 | 現場住所 | 発注者 | ステータス |
|--------|---------|--------|----------|
| 新宿駅前ビル新築工事 | 東京都新宿区西新宿一丁目1-1 | 大成建設株式会社 | 進行中 |
| 横浜港北マンション改修工事 | 神奈川県横浜市港北区新横浜二丁目3-10 | 清水建設株式会社 | 進行中 |
| 川崎工業団地倉庫建設 | 神奈川県川崎市川崎区浮島町10-5 | 鹿島建設株式会社 | 準備中 |

#### 1-2. インポート手順

1. Bitable上で「案件マスタ」テーブルを開く
2. テーブル右上の「...」（三点メニュー）をクリックする
3. 「CSVをインポート」を選択する
4. `master-projects.csv` ファイルを選択してアップロードする
5. カラムマッピング画面が表示されるので、以下の対応を確認する

   | CSVの列名 | Bitableフィールド | 対応方法 |
   |----------|-----------------|---------|
   | 案件ID | （スキップ） | 自動番号フィールドのため、インポート対象外にする |
   | 案件名 | `fld_project_name` | テキストとしてマッピング |
   | 現場住所 | `fld_site_address` | テキストとしてマッピング |
   | 現場GPS緯度 | `fld_site_lat` | 数値としてマッピング |
   | 現場GPS経度 | `fld_site_lng` | 数値としてマッピング |
   | GPS許容半径(m) | `fld_gps_radius` | 数値としてマッピング |
   | 発注者 | `fld_client` | テキストとしてマッピング |
   | 工期開始 | `fld_start_date` | 日付としてマッピング（YYYY/MM/DD形式） |
   | 工期終了 | `fld_end_date` | 日付としてマッピング（YYYY/MM/DD形式） |
   | ステータス | `fld_project_status` | 単一選択としてマッピング |
   | 現場監督 | （スキップ） | ユーザーフィールドはCSVインポート不可。後で手動設定 |
   | 備考 | `fld_project_note` | テキストとしてマッピング |

6. 「インポート」ボタンをクリックする
7. インポート結果を確認する -- **3件のレコード**が作成されていること

#### 1-3. インポート後の手動設定

以下のフィールドはCSVインポートでは設定できないため、手動で設定する。

| フィールド | 設定内容 | 手順 |
|-----------|---------|------|
| 案件ID | 自動番号 | 自動生成されるため対応不要。ただしプレフィックスが `PJ-` で開始番号が `001` であることを確認する |
| 現場監督 | ユーザー型 | 各レコードを開き、「現場監督」フィールドでLarkユーザーを検索・選択する |

> **補足**: 自動番号フィールドの値はBitableが自動で割り振るため、CSVの「PJ-001」などの値とは異なる番号になる場合があります。自動番号はあくまで一意の識別子として使用するため、番号の一致は必須ではありません。

#### 3-2. インポート手順

1. Bitable上で「作業員マスタ」テーブルを開く
2. テーブル右上の「...」（三点メニュー）をクリックする
3. 「CSVをインポート」を選択する
4. `master-workers.csv` ファイルを選択してアップロードする
5. カラムマッピング画面で以下の対応を確認する

   | CSVの列名 | Bitableフィールド | 対応方法 |
   |----------|-----------------|---------|
   | 作業員ID | （スキップ） | 自動番号フィールドのため、インポート対象外にする |
   | 氏名 | `fld_worker_name` | テキストとしてマッピング |
   | Larkアカウント | （スキップ） | ユーザーフィールドはCSVインポート不可。後で手動設定 |
   | 職種 | `fld_job_type` | 単一選択としてマッピング |

6. 「インポート」ボタンをクリックする
7. インポート結果を確認する -- **10件のレコード**が作成されていること

#### 3-3. インポート後の手動設定（必須）

作業員マスタには、CSVインポートでは設定できない3つのフィールドがあります。**必ず手動で設定してください。**

##### Larkアカウントの紐付け

1. 各作業員のレコードを開く
2. 「Larkアカウント」フィールドをクリックする
3. 該当する作業員のLarkアカウント名で検索し、選択する
4. 10名分すべてのアカウントを紐付ける

> **重要**: Larkアカウントの紐付けが未完了の場合、出退勤打刻フォームや日報フォームで作業員の自動識別ができません。必ず全員分を設定してください。

##### 所属案件のリンク設定

1. 各作業員のレコードを開く
2. 「所属案件」フィールドの「+」ボタンをクリックする
3. リンク先の「案件マスタ」テーブルから該当する案件を選択する
4. サンプルデータの場合、以下のように設定する

   | 作業員 | リンクする案件 |
   |--------|-------------|
   | 山田一郎、中村健太、小林正義、高橋勇気 | 新宿駅前ビル新築工事 |
   | 加藤修一、吉田裕介、渡辺大輔、伊藤誠 | 横浜港北マンション改修工事 |
   | 松本龍二、井上拓也 | 川崎工業団地倉庫建設 |

> **補足**: 1人の作業員が複数の案件に所属する場合は、複数の案件を選択できます。「+」ボタンで追加してください。

##### 班長フラグの確認

CSVインポートで `TRUE` / `FALSE` の値がチェックボックスに正しく反映されていることを確認する。

- 山田一郎: チェックON（班長）
- 加藤修一: チェックON（班長）
- その他: チェックOFF

> **注意**: CSVの値が文字列 `TRUE` として取り込まれ、チェックボックスがONにならないケースがあります。その場合は手動でチェックを入れてください。

## 方法2: 手動入力

データ件数が少ない場合や、CSVインポートがうまくいかない場合は、手動でデータを入力します。

### 案件マスタの手動入力

1. 「案件マスタ」テーブルを開く
2. テーブル末尾の「+」ボタンをクリックして新規レコードを追加する
3. 各フィールドに値を入力する

   **入力例（1件目）:**

   | フィールド | 入力値 |
   |-----------|--------|
   | 案件名 | 新宿駅前ビル新築工事 |
   | 現場住所 | 東京都新宿区西新宿一丁目1-1 |
   | 現場GPS緯度 | 35.689487 |
   | 現場GPS経度 | 139.700162 |
   | GPS許容半径(m) | 500 |
   | 発注者 | 大成建設株式会社 |
   | 工期開始 | 2025/10/01 |
   | 工期終了 | 2026/09/30 |
   | ステータス | 進行中 |
   | 現場監督 | （Larkユーザーを検索して選択） |
   | 備考 | 地上12階建てオフィスビル新築。夜間作業あり。 |

4. 残りの案件も同様に入力する

> **GPS座標の調べ方**: Googleマップで現場住所を検索し、地図上でピンの位置を右クリック（PCの場合）すると、緯度・経度が表示されます。その値をコピーして貼り付けてください。

### 作業内容マスタの手動入力

1. 「作業内容マスタ」テーブルを開く
2. テーブル末尾の「+」ボタンをクリックして新規レコードを追加する
3. 各フィールドに値を入力する

   **入力例（1件目）:**

   | フィールド | 入力値 |
   |-----------|--------|
   | 作業大分類 | 仮設 |
   | 作業名 | 足場組立 |
   | 標準人工 | 3.0 |

4. 必要な作業項目をすべて入力する（サンプルでは全20件）

> **補足**: 作業内容マスタは初期設定後にほとんど変更が発生しないテーブルです。自社の実態に合わせた作業項目を漏れなく登録してください。

### 作業員マスタの手動入力

1. 「作業員マスタ」テーブルを開く
2. テーブル末尾の「+」ボタンをクリックして新規レコードを追加する
3. 各フィールドに値を入力する

   **入力例（1件目）:**

   | フィールド | 入力値 |
   |-----------|--------|
   | 氏名 | 山田一郎 |
   | Larkアカウント | （Larkユーザーを検索して選択） |
   | 職種 | とび |
   | 所属班 | A班 |
   | 班長フラグ | ON（チェック） |
   | 日当単価(円) | 22000 |
   | 連絡先 | 090-1234-5678 |
   | ステータス | 稼働中 |
   | 所属案件 | （案件マスタから「新宿駅前ビル新築工事」を選択） |

4. 残りの作業員も同様に入力する

> **補足**: 手動入力の場合は、Larkアカウントや所属案件のリンクもその場で設定できるため、インポート後の手動設定が不要になります。

## インポート後の確認チェックリスト

すべてのマスタデータ投入が完了したら、以下の項目を確認してください。

### レコード数の確認

- [ ] 案件マスタ: 3件（またはカスタマイズした件数）のレコードが存在する
- [ ] 作業内容マスタ: 20件（またはカスタマイズした件数）のレコードが存在する
- [ ] 作業員マスタ: 10件（またはカスタマイズした件数）のレコードが存在する

### フィールド値の確認

- [ ] 案件マスタの「ステータス」が正しい選択肢（準備中 / 進行中 / 完了 / 中止）になっている
- [ ] 案件マスタの「現場監督」にLarkユーザーが設定されている
- [ ] 案件マスタのGPS緯度・経度に正しい数値が入っている（小数点以下6桁）
- [ ] 作業内容マスタの「作業大分類」が正しい選択肢になっている
- [ ] 作業員マスタの「職種」が正しい選択肢になっている
- [ ] 作業員マスタの「ステータス」が正しい選択肢（稼働中 / 休職中 / 退職済）になっている
- [ ] 作業員マスタの「班長フラグ」が正しく設定されている（山田一郎、加藤修一がON）
- [ ] 作業員マスタの「日当単価」に正しい金額が入っている

### リンクフィールドの確認

- [ ] 作業員マスタの「所属案件」が案件マスタの正しいレコードにリンクされている
- [ ] 案件マスタ側から作業員マスタへの逆リンク（関連レコード）が表示されている

### 数式フィールドの確認

- [ ] 工数集計テーブルの金額計算式にエラーが表示されていない
- [ ] 出退勤記録テーブルの実働時間・人工数の数式にエラーが表示されていない

## 自社データへのカスタマイズガイド

サンプルデータを確認した後、実際の自社データを投入する際の注意事項です。

### 案件マスタのカスタマイズ

| 項目 | カスタマイズのポイント |
|------|---------------------|
| 案件名 | 社内の命名規則に従い正式名称を入力する |
| 現場住所 | 実際の現場住所を入力する |
| GPS座標 | Googleマップで現場の中心点を特定し、緯度・経度を入力する |
| GPS許容半径 | 現場の規模に応じて調整する。小規模現場は300m、大規模現場は1000m以上を推奨 |
| 工期 | 実際の契約工期を入力する |
| 現場監督 | 実際の現場監督のLarkアカウントを設定する |

> **GPS座標の確認方法**:
> 1. Googleマップ（https://maps.google.com/ ）を開く
> 2. 現場住所で検索する
> 3. 現場の中心付近を右クリックする
> 4. 表示される座標（例: 35.689487, 139.700162）をコピーする
> 5. カンマの前の値を「現場GPS緯度」、後の値を「現場GPS経度」に入力する

### 作業員マスタのカスタマイズ

| 項目 | カスタマイズのポイント |
|------|---------------------|
| 全作業員の登録 | 現場に出入りする全作業員を登録する。協力会社の作業員も含める |
| Larkアカウント | 全員がLarkアカウントを持っていることを確認する。未取得者は先にアカウント作成する |
| 職種の選択肢 | 既定の10職種に該当しない場合は、テーブル設計で選択肢を追加する |
| 日当単価 | 契約に基づく正確な単価を入力する。税抜の基本単価を設定する |
| 所属案件 | 複数現場を掛け持ちする作業員は、すべての案件をリンクする |

### 作業内容マスタのカスタマイズ

| 項目 | カスタマイズのポイント |
|------|---------------------|
| 作業項目の追加 | 自社特有の作業がサンプルに含まれていない場合は追加する |
| 作業項目の削除 | 自社で実施しない作業はレコードを削除するか、使用しない旨を備考に記載する |
| 大分類の追加 | 8分類で不足する場合は、テーブル設計で「作業大分類」の選択肢を追加する |
| 標準人工の調整 | 自社の実績値に基づいて標準人工を調整する。初期値は参考値として設定されている |

## トラブルシューティング

### CSVインポート時の文字化け

**症状**: インポートしたデータが文字化けしている（例: `?????` や `鐃ｿ` のような表示）

**原因**: CSVファイルのエンコーディングがUTF-8以外（Shift_JISなど）になっている

**対処法**:
1. CSVファイルをメモ帳（Windows）またはテキストエディタで開く
2. 「名前を付けて保存」を選択する
3. エンコーディングを **UTF-8（BOM付き）** に設定して保存する
4. 再度インポートを実行する

> **補足**: macOSの場合は、テキストエディットでCSVファイルを開き、「フォーマット」>「標準テキストにする」を選択した後、「ファイル」>「保存」でエンコーディングをUTF-8に変更してください。

### インポート時にフィールド型の不一致エラー

**症状**: 「データ型が一致しません」等のエラーが表示される

**原因**: CSVの値とBitableフィールドの型が一致していない

**対処法**:
1. エラーが表示されたフィールドを確認する
2. 以下のよくある不一致パターンを確認する

   | Bitableフィールド型 | よくある原因 | 対処 |
   |-------------------|------------|------|
   | 数値 | CSVの値にカンマ区切り（例: `22,000`）が含まれる | カンマを削除（例: `22000`）してから再インポート |
   | 日付 | CSV内の日付形式が `2025-10-01` など | `2025/10/01` 形式に統一する |
   | 単一選択 | CSVの値がBitableの選択肢と完全一致しない | 表記を揃える（全角/半角、スペースの有無） |
   | チェックボックス | `TRUE`/`FALSE` 以外の値（例: `1`/`0`、`はい`/`いいえ`） | `TRUE` / `FALSE` に変換する |

### リンクフィールドがCSVインポートで設定できない

**症状**: CSVの「所属案件」列（例: `PJ-001`）がリンクフィールドに反映されない

**原因**: Bitableのリンクフィールドはレコード間の関連付けで動作するため、CSVの文字列では直接設定できない

**対処法**:
1. CSVインポート時に「所属案件」列はスキップする
2. インポート完了後、各レコードを開いて手動でリンクを設定する
3. リンク設定手順:
   - 「所属案件」フィールドの「+」ボタンをクリック
   - 案件マスタのレコード一覧が表示される
   - 該当する案件を選択する

### ユーザーフィールドが設定できない

**症状**: 「Larkアカウント」や「現場監督」フィールドにCSVの値（メールアドレス等）が反映されない

**原因**: ユーザーフィールドはLarkの組織内ユーザーとの紐付けが必要であり、CSVテキストでは設定できない

**対処法**:
1. CSVインポート時に該当列はスキップする
2. インポート完了後、各レコードを開いて手動設定する
3. フィールドをクリックすると組織内ユーザーの検索窓が表示される
4. 氏名またはメールアドレスで検索して選択する

### インポート後にレコード数が合わない

**症状**: CSVの行数とインポートされたレコード数が一致しない

**原因**: CSVの末尾に空行が含まれている、またはヘッダー行の扱いが異なる

**対処法**:
1. CSVファイルを開き、末尾の空行を削除する
2. ヘッダー行（1行目）がBitableに「最初の行をヘッダーとして使用」として認識されているか確認する
3. 再度インポートを実行する

## 次のステップ

マスタデータの投入が完了したら、テスト運用を開始します。

### テスト運用の手順

1. **出退勤打刻テスト**: スマートフォンから出退勤打刻フォームを開き、サンプルの作業員で出勤・退勤の打刻を行う
2. **日報入力テスト**: 日報フォームからサンプルデータを使って日報を提出する
3. **自動化ルールの確認**: 打刻・日報提出後に、自動化ルール（通知、ステータス更新、退勤時刻の転記など）が正しく動作するかを確認する
4. **ダッシュボードの確認**: テストデータがダッシュボードのグラフ・集計に正しく反映されることを確認する
5. **承認フローの確認**: 班長承認・監督承認のフローが正しく動作するかをテストする

> **重要**: テスト運用で入力したデータは、本番運用開始前に必ず削除してください。マスタデータ（案件・作業員・作業内容）は残したまま、出退勤記録・日報のテストレコードのみを削除します。
