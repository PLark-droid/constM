# 05. 自動化ルール設定手順

建設業向け勤怠・日報テンプレート — 7つの自動化ルール作成ガイド

---

## 概要

本ガイドでは、Lark Bitable の自動化（Automation）機能を使い、7つの自動化ルールを設定します。これにより、退勤データの統合、通知、承認フロー、安全アラートを自動化し、手作業を最小限に抑えます。

**所要時間:** 約90〜120分（全7ルール）

**前提:**
- [02-create-tables.md](./02-create-tables.md) でテーブル構築が完了していること
- [03-create-views.md](./03-create-views.md) でビュー設定が完了していること
- 各作業員がLarkアカウントを保有し、作業員マスタの `Larkアカウント` フィールドに登録済みであること
- Bitableの管理者権限を持つアカウントでログインしていること

---

## 自動化ルール一覧

| # | ルール名 | トリガー | 目的 | 優先度 |
|---|---------|---------|------|--------|
| A1 | 退勤データ統合 | レコード追加（出退勤記録） | 退勤時刻を出勤レコードに転記 | 最高 |
| A2 | 日報未提出リマインダー | 定時実行（毎日18:00） | 日報未提出者へリマインド | 高 |
| A3 | 承認依頼通知 | フィールド変更（日報） | 班長に承認依頼を通知 | 高 |
| A4 | 承認フロー進行 | フィールド変更（日報） | 現場監督に最終承認を通知 | 高 |
| A5 | 差戻し通知 | フィールド変更（日報） | 報告者に差戻しを通知 | 高 |
| A6 | 長時間勤務アラート | レコード更新（出退勤記録） | 12時間超過の警告通知 | 最高 |
| A7 | GPS異常検知 | レコード追加（出退勤記録） | 現場から離れた打刻の警告 | 中 |

### 導入順序（推奨）

全ルールを一度に有効化せず、段階的に導入してください。各フェーズで1週間程度の運用テストを行い、問題がないことを確認してから次へ進みます。

| フェーズ | 対象ルール | 目的 | 期間目安 |
|---------|-----------|------|---------|
| Phase 1 | A1 | データ統合の基盤を確立 | 1週目 |
| Phase 2 | A3, A4, A5 | 承認プロセスの自動化 | 2週目 |
| Phase 3 | A2 | 入力漏れの防止 | 3週目 |
| Phase 4 | A6, A7 | 安全管理の強化 | 4週目 |

## 自動化の基本操作

すべてのルール設定で共通する操作です。初めに確認しておいてください。

### 自動化画面を開く

1. ブラウザでBitableを開く
2. 画面右上の **「自動化」ボタン**（稲妻アイコン）をクリック
3. 自動化パネルが右側に表示される

### ルールを新規作成する

1. 自動化パネル内の **「+ ルールを作成」** をクリック
2. ルール名を入力する（後で変更可能）
3. トリガーを設定する
4. 条件を設定する（任意）
5. アクションを設定する
6. **「保存」** をクリック
7. ルールのスイッチを **ON** にして有効化する

### ルールのテスト実行

1. 保存済みルールの右側にある **「...」** メニューをクリック
2. **「テスト実行」** を選択
3. テストレコードを選択して実行する
4. 実行ログで結果を確認する

### 実行ログの確認方法

1. 自動化パネル内のルールをクリック
2. **「実行ログ」** タブを選択
3. 各実行の成功/失敗と詳細を確認できる

## A1: 退勤データ統合（最重要）

### 目的

退勤フォームで作成されたレコードの退勤時刻を、同一日の出勤レコードに自動転記します。出勤と退勤を1つのレコードに統合することで、実働時間・人工数が自動計算されます。

> **重要:** このルールは全データ統合の基盤です。最初に設定し、十分にテストしてください。

### Step 1: ルールの作成

1. 自動化パネルの **「+ ルールを作成」** をクリック
2. ルール名に **「A1: 退勤データ統合」** と入力

### Step 2: トリガー設定

1. トリガーの種類で **「レコードが追加された時」** を選択
2. テーブルのドロップダウンから **「出退勤記録」** を選択
3. **「条件を追加」** をクリック
4. 左のドロップダウンから **「打刻種別」** を選択
5. 演算子が **「次の値に等しい」** であることを確認
6. 右の値に **「退勤」** を選択
7. 設定内容を確認:
   - トリガー: レコードが追加された時
   - テーブル: 出退勤記録
   - 条件: 打刻種別 = 退勤

### Step 3: アクション1 — レコード検索

退勤レコードに対応する出勤レコードを検索します。

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「レコードを検索」** を選択
3. 検索先テーブルのドロップダウンから **「出退勤記録」** を選択
4. **「検索条件を追加」** をクリックし、以下4つの条件をすべて設定する:

| # | フィールド | 演算子 | 値 |
|---|----------|--------|-----|
| 1 | 作業員 | 次の値に等しい | トリガーレコードの「作業員」 |
| 2 | 打刻日 | 次の値に等しい | トリガーレコードの「打刻日」 |
| 3 | 対象案件 | 次の値に等しい | トリガーレコードの「対象案件」 |
| 4 | 打刻種別 | 次の値に等しい | 「出勤」（固定値） |

> **値の設定方法:** 条件値の入力欄をクリックすると、「トリガーレコードのフィールド」を選択できるポップアップが表示されます。「作業員」「打刻日」「対象案件」はトリガーレコードの値を参照し、「打刻種別」は固定値「出勤」を直接入力します。

5. 検索結果の上限を **「1件」** に設定（設定可能な場合）

### Step 4: アクション2 — レコード更新

検索した出勤レコードに退勤時刻とGPS情報を転記します。

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「レコードを更新」** を選択
3. 対象レコードのドロップダウンから **「アクション1（レコード検索）の検索結果」** を選択
4. **「更新フィールドを追加」** をクリックし、以下2つのフィールドを設定する:

| # | 更新フィールド | 値 |
|---|--------------|-----|
| 1 | 退勤時刻 | トリガーレコードの「退勤時刻」 |
| 2 | 退勤GPS | トリガーレコードの「退勤GPS」 |

> **値の設定方法:** 更新値の入力欄をクリックし、「トリガーレコードのフィールド」から対応するフィールドを選択します。

### Step 5: 保存と有効化

1. 設定内容を上から通して確認する
2. **「保存」** をクリック
3. ルールのスイッチを **ON** にして有効化

### Step 6: テスト

#### 正常系テスト

1. 出勤打刻フォーム（または出退勤記録テーブルに直接）からテスト用の出勤レコードを作成する
   - 作業員: テスト作業員を選択
   - 対象案件: テスト案件を選択
   - 打刻種別: 「出勤」
   - 打刻日: 本日の日付
   - 出勤時刻: 08:00
2. 数秒待つ（自動化の実行にはわずかなタイムラグがあります）
3. 退勤打刻フォーム（または出退勤記録テーブルに直接）から退勤レコードを作成する
   - 作業員: 同一作業員を選択
   - 対象案件: 同一案件を選択
   - 打刻種別: 「退勤」
   - 打刻日: 同一日付
   - 退勤時刻: 17:00
4. 数秒待つ
5. 出勤レコードを確認する:
   - **退勤時刻** が 17:00 に更新されているか
   - **実働時間** が自動計算されているか（休憩1hの場合、8.0h）
   - **人工数** が自動計算されているか（8.0h >= 7h なので 1.0）

#### 異常系テスト

1. 出勤レコードが存在しない状態で退勤レコードを作成する
2. 確認: エラーにならず、退勤レコードのみが残ること

#### 複数案件テスト

1. 同一作業員で異なる案件の出勤レコードを2件作成する
2. 片方の案件に対して退勤レコードを作成する
3. 確認: 正しい案件の出勤レコードにのみ退勤時刻が転記されること

### 注意事項

| 注意点 | 対処方法 |
|--------|---------|
| 日付のフォーマット不一致で検索が失敗する場合がある | フォーム側で日付フォーマットを統一する |
| 同一日・同一案件に複数回出勤した場合、最新のレコードにのみ転記される | 運用ルールとして1日1案件1打刻を徹底する |
| 自動化はレコード追加の数秒後に実行される（即時ではない） | 退勤打刻後、数秒待ってからレコードを確認する |
| 地理情報フィールドの転記がBitableバージョンにより非対応の場合がある | 非対応の場合はテキストフィールドに緯度・経度を文字列として記録する代替案を検討する |

## A2: 日報未提出リマインダー

### 目的

出勤記録がある作業員に対し、日報の提出を促すリマインダー通知を毎日18:00に送信します。

> **注意:** Bitableの定時トリガーでは2つのテーブルを跨いだ差分チェック（出勤はあるが日報がない作業員の特定）を単一ルールで実現することが困難です。本ガイドでは「簡易版（全員リマインド方式）」と「フラグ方式（差分チェック対応）」の2つの実装方法を説明します。まず簡易版で運用を開始し、必要に応じてフラグ方式に移行することを推奨します。

### 方式の比較

| 項目 | 簡易版 | フラグ方式 |
|------|--------|-----------|
| 実装難易度 | 低 | 中 |
| 必要ルール数 | 1 | 3（補助ルール2 + メインルール1） |
| 通知対象 | 本日の出勤者全員 | 日報未提出者のみ |
| 利点 | 設定がシンプルで確実 | 提出済みの人に通知が届かない |
| 欠点 | 提出済みの人にも通知が届く | 設定が複雑、追加フィールドが必要 |

### 簡易版の設定手順

#### Step 1: ルールの作成

1. 自動化パネルの **「+ ルールを作成」** をクリック
2. ルール名に **「A2: 日報未提出リマインダー」** と入力

#### Step 2: トリガー設定

1. トリガーの種類で **「定時実行」** を選択
2. 実行頻度のドロップダウンから **「毎日」** を選択
3. 実行時刻を **「18:00」** に設定
4. 設定内容を確認:
   - トリガー: 定時実行
   - 頻度: 毎日
   - 時刻: 18:00

#### Step 3: アクション1 — レコード検索

本日の出勤者を検索します。

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「レコードを検索」** を選択
3. 検索先テーブルのドロップダウンから **「出退勤記録」** を選択
4. 検索条件を設定する:

| # | フィールド | 演算子 | 値 |
|---|----------|--------|-----|
| 1 | 打刻日 | 次の値に等しい | TODAY()（本日の日付） |
| 2 | 打刻種別 | 次の値に等しい | 「出勤」 |

> **TODAY() の入力方法:** 日付条件の値を設定する際、「動的な値」や「関数」のオプションが表示される場合は「TODAY」を選択します。表示されない場合は、本日の日付を直接入力してください（ただし毎日手動更新が必要になるため、動的な値の設定を推奨します）。

#### Step 4: アクション2 — 通知送信

検索結果の全出勤者に通知を送信します。

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「通知を送信」** を選択
3. 通知先の設定:
   - 検索結果のレコードの **「作業員」** フィールドを選択
   - 作業員マスタの **「Larkアカウント」** フィールドが通知先として参照される
   - ルックアップフィールドで作業員のLarkアカウントを出退勤記録に引き込んでいる場合は、そのフィールドを直接指定
4. 通知内容を入力:

```
本日の日報は提出済みですか？未提出の方は18:30までに提出をお願いします。
```

> **通知先の設定がうまくいかない場合:** 出退勤記録テーブルにルックアップフィールド「作業員Larkアカウント」を追加し、作業員マスタの `Larkアカウント` フィールドを引き込んでおくと、通知先として直接指定できます。

5. **「保存」** をクリック
6. ルールのスイッチを **ON** にして有効化

#### 簡易版のテスト

1. 出退勤記録テーブルに本日日付の出勤レコードを作成する
2. 自動化ルールの **「テスト実行」** ボタンをクリック
3. 確認: 出勤レコードの作業員にLark通知が届くこと
4. 確認: 通知メッセージの内容が正しいこと

### フラグ方式の設定手順（高度）

差分チェックにより日報未提出者のみに通知する方式です。

#### 事前準備: フィールド追加

1. 出退勤記録テーブルを開く
2. 右端の **「+」** ボタンからフィールドを追加する:
   - フィールド名: **「日報提出済」**
   - 型: **チェックボックス**
   - デフォルト値: **OFF（未チェック）**
3. **「確定」** をクリック

#### 補助ルール A2-a: 日報提出フラグ更新

日報が提出された時に、対応する出退勤レコードのフラグをONにします。

1. 自動化パネルの **「+ ルールを作成」** をクリック
2. ルール名に **「A2-a: 日報提出フラグ更新」** と入力
3. トリガー設定:
   - 種類: **「レコードが追加された時」**
   - テーブル: **「日報」**
4. アクション1 — レコード検索:
   - 種類: **「レコードを検索」**
   - 検索先テーブル: **「出退勤記録」**
   - 検索条件:

| # | フィールド | 演算子 | 値 |
|---|----------|--------|-----|
| 1 | 作業員 | 次の値に等しい | トリガーレコードの「報告者」 |
| 2 | 打刻日 | 次の値に等しい | トリガーレコードの「報告日」 |
| 3 | 打刻種別 | 次の値に等しい | 「出勤」 |

5. アクション2 — レコード更新:
   - 種類: **「レコードを更新」**
   - 対象レコード: アクション1の検索結果
   - 更新フィールド: **「日報提出済」** = **ON（チェック）**
6. **「保存」** をクリックし、スイッチを **ON** にして有効化

#### 補助ルール A2-b: 定時リマインダー（未提出者限定）

1. 自動化パネルの **「+ ルールを作成」** をクリック
2. ルール名に **「A2-b: 日報未提出リマインダー（フラグ方式）」** と入力
3. トリガー設定:
   - 種類: **「定時実行」**
   - 頻度: **「毎日」**
   - 時刻: **「18:00」**
4. アクション1 — レコード検索:
   - 種類: **「レコードを検索」**
   - 検索先テーブル: **「出退勤記録」**
   - 検索条件:

| # | フィールド | 演算子 | 値 |
|---|----------|--------|-----|
| 1 | 打刻日 | 次の値に等しい | TODAY() |
| 2 | 打刻種別 | 次の値に等しい | 「出勤」 |
| 3 | 日報提出済 | 次の値に等しい | OFF（未チェック） |

5. アクション2 — 通知送信:
   - 種類: **「通知を送信」**
   - 通知先: 検索結果の作業員のLarkアカウント
   - 通知内容:

```
本日の日報が未提出です。18:30までに提出してください。
```

6. **「保存」** をクリックし、スイッチを **ON** にして有効化

#### フラグ方式のテスト

1. 出退勤記録テーブルに本日日付の出勤レコードを作成する（日報提出済 = OFF）
2. 日報テーブルに該当作業員の日報を作成する
3. 確認: 出退勤記録の「日報提出済」が **ON** に更新されること（A2-aの動作確認）
4. 自動化ルールA2-bを **「テスト実行」** する
5. 確認: 日報提出済 = OFF のレコードの作業員のみに通知が届くこと

### 注意事項

| 注意点 | 対処方法 |
|--------|---------|
| 定時トリガーの実行時刻は数分ずれる場合がある | 数分の誤差を許容する運用とする |
| 通知先のLarkアカウント解決が複雑 | ルックアップフィールドでLarkアカウントを出退勤記録に引き込んでおく |
| 簡易版では提出済みの作業員にも通知が届く | 通知メッセージに「提出済みの方は無視してください」と追記する運用も可 |

## A3: 承認依頼通知

### 目的

日報が提出されたことを班長に通知し、承認作業を促します。

### Step 1: ルールの作成

1. 自動化パネルの **「+ ルールを作成」** をクリック
2. ルール名に **「A3: 承認依頼通知」** と入力

### Step 2: トリガー設定

1. トリガーの種類で **「レコードが更新された時」** を選択
2. テーブルのドロップダウンから **「日報」** を選択
3. 監視フィールドの設定:
   - **「監視するフィールドを選択」** をクリック
   - **「承認状態」** にチェックを入れる
4. **「条件を追加」** をクリック
5. 左のドロップダウンから **「承認状態」** を選択
6. 演算子が **「次の値に等しい」** であることを確認
7. 右の値に **「提出済」** を選択
8. 設定内容を確認:
   - トリガー: レコードが更新された時
   - テーブル: 日報
   - 監視フィールド: 承認状態
   - 条件: 承認状態 = 提出済

### Step 3: アクション1 — 班長を検索

承認担当の班長を検索します。

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「レコードを検索」** を選択
3. 検索先テーブルのドロップダウンから **「作業員マスタ」** を選択
4. 検索条件を設定する:

| # | フィールド | 演算子 | 値 |
|---|----------|--------|-----|
| 1 | 班長フラグ | 次の値に等しい | ON（チェック済み） |
| 2 | ステータス | 次の値に等しい | 「稼働中」 |

> **同一班の班長に限定する場合:** 報告者の所属班と同一の班長のみを通知先にしたい場合は、検索条件に「所属班 = トリガーレコードの報告者の所属班」を追加します。ただし、リンクフィールド経由のクロステーブル条件はBitableのバージョンにより設定が困難な場合があります。その場合は、日報テーブルにルックアップフィールド「報告者所属班」を追加し、それを条件に使用してください。

### Step 4: アクション2 — 通知送信

## A6: 長時間勤務アラート

### 目的

実働時間が12時間を超える長時間勤務を検知し、班長と現場監督に警告通知を送信します。労働安全衛生の観点から重要なルールです。

> **重要:** このルールはA1（退勤データ統合）が正常に動作していることが前提です。A1で退勤時刻が転記された結果、実働時間が計算され、その値に基づいてアラートが発動します。

### 数式フィールドの条件利用について

Bitableの自動化では、数式フィールド（`実働時間`）の値を条件に使用できる場合とできない場合があります。以下に2つの方法を示します。

| 方法 | 条件の指定方法 | 適用場面 |
|------|-------------|---------|
| 方法A（推奨） | 「長時間フラグ」チェックボックスを追加 | 数式フィールドが条件に使用できない場合 |
| 方法B | 実働時間フィールドを直接条件に使用 | 数式フィールドが条件に使用できる場合 |

まず方法Bを試し、うまくいかない場合は方法Aを採用してください。

### 事前準備（方法Aの場合のみ）

数式フィールドを条件に使用できない場合、フラグ用フィールドを追加します。

1. 出退勤記録テーブルを開く
2. 右端の **「+」** ボタンからフィールドを追加する:
   - フィールド名: **「長時間フラグ」**
   - 型: **数式**
   - 数式: `IF(fld_actual_hours > 12, true, false)`

> **注意:** ここでの `fld_actual_hours` はフィールドIDです。Bitableの数式エディタでは、実際のフィールド名（「実働時間(h)」）をクリックして挿入します。

3. 監視フィールドの設定:

4. **「条件を追加」** をクリック

**方法A（長時間フラグ使用）の場合:**
- 条件: **「長時間フラグ」** = **true**

**方法B（実働時間フィールド直接使用）の場合:**

6. 設定内容を確認:
   - トリガー: レコードが更新された時

| 注意点 | 対処方法 |
|--------|---------|

## A7: GPS異常検知

### 目的

出勤打刻位置が現場から離れている場合に管理者に通知します。不正打刻の防止と、直行・直帰時の把握に役立ちます。

> **注意:** GPS異常検知はBitableの標準機能だけでは完全な実装が困難な場合があります。本ガイドでは、数式フィールドによる簡易距離計算方式（方式A）を説明します。

### 距離計算の考え方

Bitableの数式フィールドでは、完全なHaversine公式（球面三角法）の実装は困難です。日本の緯度帯（北緯30度〜45度）で有効な簡易近似式を使用します。

**近似距離計算式:**

```
距離(m) = SQRT(
  ((打刻緯度 - 現場緯度) * 111000) ^ 2 +
  ((打刻経度 - 現場経度) * 111000 * COS(RADIANS((打刻緯度 + 現場緯度) / 2))) ^ 2
)
```

- `111000`: 緯度1度あたりの距離（約111km）をメートルに換算した値
- `COS(RADIANS(...))`: 経度方向の補正係数（日本の緯度帯で約0.77〜0.87）
- GPS許容半径が数百メートルの判定であれば、実用上十分な精度

### 事前準備: フィールド追加

GPS異常検知には、座標値を数値として扱うための追加フィールドが必要です。

#### 出退勤記録テーブルにフィールドを追加

1. 出退勤記録テーブルを開く
2. 右端の **「+」** ボタンから以下のフィールドを順番に追加する:

| # | フィールド名 | 型 | 設定 |
|---|-------------|-----|------|
| 1 | 打刻緯度 | 数値 | 小数点以下6桁 |
| 2 | 打刻経度 | 数値 | 小数点以下6桁 |

> **座標値の取得方法:** 地理情報フィールド（出勤GPS）から緯度・経度を数値として自動抽出できない場合は、出勤フォームに緯度・経度の入力欄を追加するか、JavaScriptウィジェットやLark Flowで座標値を抽出する方法を検討してください。

3. ルックアップフィールドを追加する:

| # | フィールド名 | 型 | 参照先 |
|---|-------------|-----|--------|
| 3 | 現場緯度（ルックアップ） | ルックアップ | 案件マスタの「現場GPS緯度」 |
| 4 | 現場経度（ルックアップ） | ルックアップ | 案件マスタの「現場GPS経度」 |
| 5 | GPS許容半径（ルックアップ） | ルックアップ | 案件マスタの「GPS許容半径(m)」 |

各ルックアップフィールドの作成手順:
1. **「+」** → 型を **「ルックアップ」** に設定
2. 参照元フィールドとして **「対象案件」**（リンクフィールド）を選択
3. 参照先テーブルの対応するフィールド（現場GPS緯度 等）を選択
4. **「確定」** をクリック

4. 数式フィールドを追加する:

| # | フィールド名 | 型 | 数式 |
|---|-------------|-----|------|
| 6 | 現場距離(m) | 数式 | 下記参照 |
| 7 | GPS異常フラグ | 数式 | 下記参照 |

**現場距離(m) の数式:**

```
SQRT(
  POWER((打刻緯度 - 現場緯度（ルックアップ）) * 111000, 2) +
  POWER((打刻経度 - 現場経度（ルックアップ）) * 111000 *
    COS(RADIANS((打刻緯度 + 現場緯度（ルックアップ）) / 2)), 2)
)
```

> **数式の入力方法:** Bitableの数式エディタでは、フィールド名をクリックして挿入します。上記の数式内のフィールド名は日本語表示名です。数式エディタでフィールド名をクリックすると、自動的に正しい参照形式に変換されます。

**GPS異常フラグ の数式:**

```
IF(AND(現場距離(m) > 0, 現場距離(m) > GPS許容半径（ルックアップ）), true, false)
```

> **AND条件の意味:** `現場距離(m) > 0` は、座標が未入力の場合（距離 = 0 または空）にアラートが誤発動しないようにするためのガード条件です。

### Step 1: ルールの作成

1. 自動化パネルの **「+ ルールを作成」** をクリック
2. ルール名に **「A7: GPS異常検知」** と入力

### Step 2: トリガー設定

1. トリガーの種類で **「レコードが追加された時」** を選択
2. テーブルのドロップダウンから **「出退勤記録」** を選択
3. **「条件を追加」** をクリック
4. 条件を設定する:

| # | フィールド | 演算子 | 値 |
|---|----------|--------|-----|
| 1 | 打刻種別 | 次の値に等しい | 「出勤」 |
| 2 | GPS異常フラグ | 次の値に等しい | true |

> **数式フィールドを条件に使用できない場合:** GPS異常フラグが条件として認識されない場合は、トリガー条件を「打刻種別 = 出勤」のみにし、アクション側でレコード検索を使って「GPS異常フラグ = true」のレコードをフィルタリングする方法を検討してください。

5. 設定内容を確認:
   - トリガー: レコードが追加された時
   - テーブル: 出退勤記録
   - 条件: 打刻種別 = 出勤 AND GPS異常フラグ = true

### Step 3: アクション1 — 現場監督を検索

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「レコードを検索」** を選択
3. 検索先テーブルのドロップダウンから **「案件マスタ」** を選択
4. 検索条件: トリガーレコードの **「対象案件」** のリンク先レコードを指定

### Step 4: アクション2 — 管理者に警告通知

1. **「+ アクションを追加」** をクリック
2. アクションの種類で **「通知を送信」** を選択
3. 通知先: アクション1の検索結果の **「現場監督」** フィールドを選択
4. 通知内容を入力:

```
[警告] {作業員}さんの出勤打刻位置が現場から離れています。確認してください。
日付: {打刻日}
案件: {対象案件}
現場からの距離: 約{現場距離(m)}m
許容半径: {GPS許容半径（ルックアップ）}m
```

5. **「保存」** をクリック
6. ルールのスイッチを **ON** にして有効化

### テスト

#### 異常検知テスト（アラート発動）

1. 案件マスタにテスト案件のGPS座標を設定する
   - 現場GPS緯度: `35.681200`
   - 現場GPS経度: `139.767100`
   - GPS許容半径(m): `500`
2. 出退勤記録テーブルに出勤レコードを作成する
   - 打刻種別: 「出勤」
   - 対象案件: テスト案件
   - 打刻緯度: `35.690000`
   - 打刻経度: `139.780000`
   - (現場から約1.6km離れた座標 → 閾値超過)
3. 確認: GPS異常フラグが **true** になること
4. 確認: 現場監督にLark警告通知が届くこと

#### 正常範囲テスト（アラート非発動）

1. 出退勤記録テーブルに出勤レコードを作成する
   - 打刻緯度: `35.681500`
   - 打刻経度: `139.767500`
   - (現場から約50m → 閾値以内)
2. 確認: GPS異常フラグが **false** になること
3. 確認: 通知が送信されないこと

#### GPS未入力テスト

1. 打刻緯度・打刻経度が空の出勤レコードを作成する
2. 確認: エラーにならないこと（距離 = 0 となり、アラートは発動しない）

### 注意事項

| 注意点 | 対処方法 |
|--------|---------|
| 地理情報フィールドから緯度・経度を数値として取得できない場合がある | フォームで緯度・経度を数値フィールドとして別途取得する |
| 案件マスタにGPS座標が未設定の場合は距離計算ができない | GPS座標未設定の案件ではアラートがスキップされる（AND条件でガード済み） |
| 屋内・地下ではGPS精度が低下する | 許容半径を広めに設定する（例: 1000m） |
| 直行・直帰の場合は現場以外からの打刻が正当 | 出勤ステータスが「直行」の場合はアラートを除外する条件を追加する |
| 簡易距離計算の精度限界 | 許容半径が数百メートルの判定では十分な精度。高精度が必要な場合はWebhook方式（外部サーバーでHaversine公式を計算）を検討する |

## トラブルシューティング

### 自動化が発火しない場合

| 確認項目 | 対処方法 |
|---------|---------|
| ルールのスイッチがONになっているか | 自動化パネルでスイッチの状態を確認する |
| トリガー条件が正しいか | トリガーの条件値（「退勤」「出勤」「提出済」等）がテーブルの選択肢と完全一致しているか確認する。全角・半角、スペースの有無に注意 |
| トリガーの種類が正しいか | 「レコード追加」と「レコード更新」を間違えていないか確認する。既存レコードのフィールド値を変更しても「レコード追加」トリガーは発火しない |
| 監視フィールドが正しく設定されているか | 「レコード更新」トリガーでは、監視対象のフィールドが変更された時のみ発火する。監視フィールドの設定を確認する |
| 月間実行回数の上限に達していないか | Bitableの管理画面で当月の実行回数を確認する。プランにより上限が異なる |
| 自動化機能がプランで利用可能か | 利用中のLarkプランで自動化機能が含まれているか確認する |

### 通知が届かない場合

| 確認項目 | 対処方法 |
|---------|---------|
| 通知先のLarkアカウントが正しく設定されているか | 作業員マスタの「Larkアカウント」フィールドに正しいアカウントが登録されているか確認する |
| 検索結果が空ではないか | 自動化の実行ログを確認し、レコード検索のステップで結果が0件になっていないか確認する |
| 通知先のフィールド参照が正しいか | 通知先に指定したフィールドがユーザー型またはLarkアカウントを含むフィールドか確認する |
| Larkの通知設定が有効か | 受信者側のLarkアプリで通知がオフになっていないか、おやすみモードになっていないか確認する |
| リンクフィールド経由の参照が機能しているか | ルックアップフィールドでLarkアカウントを事前に引き込んでいるか確認する |

### 検索結果が空の場合

| 確認項目 | 対処方法 |
|---------|---------|
| 検索条件の値が一致しているか | 検索条件に使用しているフィールドの値が、対象レコードの値と完全一致しているか確認する |
| 日付フォーマットが一致しているか | 検索条件の日付とレコードの日付のフォーマット（YYYY/MM/DD 等）が同一か確認する |
| リンクフィールドの参照が正しいか | 「トリガーレコードの作業員」と検索対象の「作業員」が同じリンク先を参照しているか確認する |
| テストデータが存在するか | 検索対象のテーブルに条件に合致するレコードが存在するか直接確認する |
| トリガーレコードのフィールド値を確認する | 自動化の実行ログで、トリガーレコードの各フィールド値が想定通りか確認する |

### A1固有のトラブル

| 症状 | 原因と対処方法 |
|------|-------------|
| 退勤時刻が転記されない | 検索条件の4つ（作業員・打刻日・対象案件・打刻種別）がすべて正しく設定されているか確認する。1つでも不一致があると出勤レコードが見つからない |
| 別の案件の出勤レコードに転記された | 検索条件に「対象案件」が含まれているか確認する |
| 実働時間が計算されない | 出勤レコードに出勤時刻と退勤時刻の両方が入力された状態で、数式フィールド（実働時間）が正しく設定されているか確認する |
