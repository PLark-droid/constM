## 1. 出退勤打刻のよくある質問

### Q1-1: 出勤打刻を忘れました

**A:** 班長または現場監督に連絡してください。管理者がBitableのテーブルから直接レコードを追加できます。

**対処手順（管理者向け）:**

1. Bitableで「T4: 出退勤記録」テーブルを開く
2. 「+ レコード追加」をクリック
3. 以下の項目を入力する
   - 作業員: 該当する作業員を選択
   - 対象案件: 該当する案件を選択
   - 打刻種別: 「出勤」を選択
   - 打刻日: 該当日の日付を入力
   - 出勤時刻: 実際の出勤時刻を入力
   - 承認状態: 「未承認」のまま
4. レコードを保存する

> **注意:** 管理者が追加した場合、出勤GPSは記録されません。備考欄に「打刻忘れのため手動追加」等と記載してください。

---

### Q1-2: 退勤打刻を忘れました

**A:** 班長に連絡してください。対処方法は2つあります。

**方法1: 退勤フォームから翌日に打刻（推奨）**

翌朝、退勤打刻フォーム（F2）から前日分として送信します。ただし打刻日が自動で「今日」になるため、以下に注意してください。

- 退勤打刻フォームの打刻日は自動入力（TODAY()）のため、日付が異なる場合は自動化A1（退勤データ統合）が正しく動作しません
- この場合は方法2を使ってください

**方法2: 管理者が出勤レコードの退勤時刻を直接編集**

1. Bitableで「T4: 出退勤記録」テーブルを開く
2. 該当する出勤レコードを探す
3. `fld_clock_out`（退勤時刻）に実際の退勤時刻を入力する
4. 実働時間（`fld_actual_hours`）と人工数（`fld_manday`）は自動で再計算される

---

### Q1-3: 出勤時刻を間違えて打刻しました

**A:** 班長が出退勤記録テーブルから該当レコードの時刻を修正できます。

**対処手順:**

1. 班長がBitableで「T4: 出退勤記録」テーブルを開く
2. 該当レコードの`fld_clock_in`（出勤時刻）を正しい時刻に修正する
3. 保存する

> **注意:** 出勤時刻を修正すると、実働時間（`fld_actual_hours`）と人工数（`fld_manday`）が自動的に再計算されます。修正後に値が正しいことを確認してください。

---

### Q1-4: 1日に2つの現場に行きました。どう打刻すればいいですか？

**A:** 現場ごとに出退勤打刻を行ってください。それぞれ別のレコードとして記録されます。

**打刻の流れ:**

```
1つ目の現場:
  出勤打刻（名前 + 現場A + 時刻）  -->  退勤打刻（名前 + 現場A + 時刻）

2つ目の現場:
  出勤打刻（名前 + 現場B + 時刻）  -->  退勤打刻（名前 + 現場B + 時刻）
```

> **ポイント:** 自動化A1（退勤データ統合）は「作業員 + 打刻日 + 対象案件」の組み合わせで出勤レコードを検索するため、案件が異なれば正しく動作します。

---

### Q1-5: 直行・直帰の場合はどうすればいいですか？

**A:** 出勤打刻フォーム（F1）の「出勤ステータス」を変更してください。

| 状況 | 出勤ステータスの設定 |
|------|-------------------|
| 自宅から直接現場へ | 「直行」 |
| 現場から直接帰宅 | 「直帰」 |
| 自宅→現場→帰宅（事務所を経由しない） | 「直行直帰」 |

> **GPS位置について:** 出勤ステータスが「直行」の場合、GPS位置が現場と異なっていても問題ありません。ただし自動化A7（GPS異常検知）が有効な場合、管理者に通知が届く可能性があります。管理者は「直行」であることを確認してください。

---

### Q1-6: 休日出勤の打刻方法は？

**A:** 通常通り出退勤打刻をしてください。出勤ステータスを「休日出勤」に変更してください。

**手順:**

1. 出勤打刻フォーム（F1）を開く
2. 名前・現場・時刻を入力
3. 「出勤ステータス」を「休日出勤」に変更
4. 送信

---

### Q1-7: GPS位置情報が取得できません

**A:** 以下を順番に確認してください。

**確認ステップ:**

1. **スマホの位置情報サービスがONか確認**
   - iPhone: 設定 > プライバシーとセキュリティ > 位置情報サービス > ON
   - Android: 設定 > 位置情報 > ON

2. **Larkアプリの位置情報権限を確認**
   - iPhone: 設定 > Lark > 位置情報 > 「アプリ使用中に許可」
   - Android: 設定 > アプリ > Lark > 権限 > 位置情報 > 「アプリの使用中のみ許可」

3. **建物内で電波が弱い場合**
   - 屋外に出てから再試行してください
   - Wi-Fiに接続すると精度が向上する場合があります

4. **それでも取得できない場合**
   - GPS位置は必須項目ではありません。位置情報なしで送信してもエラーにはなりません
   - 備考欄に「GPS取得不可」と記載しておくと管理者が把握しやすくなります

---

### Q1-8: 同じ日に出勤を2回打刻してしまいました

**A:** 班長に連絡してください。余分なレコードを削除してもらいます。

**対処手順（管理者向け）:**

1. 「T4: 出退勤記録」テーブルで該当作業員・該当日のレコードをフィルタ
2. 重複レコードを確認
3. 不要なレコードを削除

> **注意:** 退勤打刻が既に統合されている場合は、正しい出勤レコードに退勤時刻が記録されていることを確認してから削除してください。

---

### Q1-9: 「遅刻」扱いの打刻方法は？

**A:** 出勤打刻フォーム（F1）の「出勤ステータス」を「遅刻」に変更してから送信してください。出勤時刻はそのまま現在時刻（実際の到着時刻）で打刻します。

---

### Q1-10: 天候の入力は必要ですか？

**A:** 任意項目です。入力しなくても出勤打刻は完了します。ただし、雨天や強風など現場の安全管理に関わる天候の場合は記録をお勧めします。

## 2. 日報のよくある質問

### Q2-1: 日報の書き方がわかりません

**A:** 最低限、以下の項目を入力すればOKです。

```
必須入力:
  (1) 自分の名前    --> リストから選ぶ
  (2) 現場名        --> リストから選ぶ
  (3) 作業内容1     --> メインの作業をリストから選ぶ
  (4) 作業時間1     --> 数値で入力（例: 7.0）

任意（入力推奨）:
  - 進捗メモ: 場所 + 作業 + どこまで終わったか
  - 安全事項: ヒヤリハット・危険な箇所
  - 写真: 作業完了箇所・問題箇所

送信で完了!
```

**進捗メモの書き方例:**

| 良い例 | 理由 |
|--------|------|
| 「3F 鉄筋組立 80%完了」 | 場所・作業・進捗が明確 |
| 「外壁タイル貼り 南面 完了」 | 場所が具体的 |
| 「1F配管 トイレ周り 50%」 | 進捗率がわかる |

---

### Q2-2: 日報を提出し忘れました

**A:** 翌日でも提出可能です。

**手順:**

1. 日報提出フォーム（F3）を開く
2. 通常通り入力
3. 「報告日」を **実際の作業日** に変更する（デフォルトは今日の日付）
4. 送信

> **リマインダーについて:** 毎日18:00ごろに日報未提出リマインダー（自動化A2）がLark通知として届きます。通知が届いたら、まだ日報を出していないか確認してください。

---

### Q2-3: 提出した日報を修正したい

**A:** 承認状態により対応が異なります。

| 承認状態 | 修正方法 |
|---------|---------|
| 提出済（未承認） | 班長に差戻しを依頼 --> 修正して再提出 |
| 班長承認済み | 班長に差戻しを依頼 --> 修正して再提出 |
| 監督承認済み | 管理者に直接修正を依頼 |

**差戻し後の再提出手順:**

1. 差戻し通知のコメントを確認する
2. Bitableのビューから該当日報レコードを開く
3. 指摘された箇所を修正する
4. 承認状態を「提出済」に変更する（または管理者に変更を依頼する）

---

### Q2-4: 写真を添付できません

**A:** 以下を確認してください。

1. **Larkアプリのカメラ権限を確認**
   - iPhone: 設定 > Lark > カメラ > ON
   - Android: 設定 > アプリ > Lark > 権限 > カメラ > 許可

2. **写真ライブラリの権限を確認**
   - iPhone: 設定 > Lark > 写真 > 「すべての写真」または「選択した写真」
   - Android: 設定 > アプリ > Lark > 権限 > ストレージ > 許可

3. **ファイルサイズが大きすぎる場合**
   - 高解像度の写真はファイルサイズが大きくなります
   - スマホのカメラ設定で解像度を下げるか、撮影後にリサイズしてから添付してください

4. **対応ファイル形式**
   - JPG, PNG, HEIC（iPhoneの標準形式）に対応しています
   - 動画ファイルは添付できません

---

### Q2-5: 作業内容の選択肢に該当するものがありません

**A:** 以下の手順で対応してください。

1. 選択肢の中から **一番近いもの** を選ぶ
2. 「進捗メモ」に実際の作業内容の詳細を記入する
3. 管理者に連絡し、「T3: 作業内容マスタ」への追加を依頼する

**管理者向け -- 作業内容マスタへの追加手順:**

1. 「T3: 作業内容マスタ」テーブルを開く
2. 「+ レコード追加」をクリック
3. 作業大分類（仮設/土工/基礎/躯体/仕上/設備/外構/その他）を選択
4. 作業名を入力（例:「ケレン作業」）
5. 保存する

---

### Q2-6: 作業時間の入力がよくわかりません

**A:** 0.5時間刻みで、実働時間の合計になるように入力してください。

```
例1: 1つの作業を1日やった場合
  作業1: 鉄筋組立 = 7.0 時間

例2: 2つの作業をやった場合
  作業1: 型枠建込 = 4.0 時間
  作業2: 配筋検査 = 3.0 時間

例3: 3つの作業をやった場合
  作業1: 足場組立 = 3.0 時間
  作業2: 外壁下地 = 2.5 時間
  作業3: 片付け   = 1.5 時間
```

> **ポイント:** 作業時間の合計は出退勤の実働時間と一致させてください。例えば実働7時間なら、作業1 + 作業2 + 作業3 = 7.0時間になるように入力します。

---

### Q2-7: 日報提出後、18:00のリマインダーが届きます

**A:** 自動化A2（日報未提出リマインダー）の方式によります。

- **簡易版（全員リマインド方式）の場合:** 日報提出済みでもリマインダーが届きます。提出済みであれば無視してください。
- **フラグ方式の場合:** 日報提出済みの作業員にはリマインダーは届きません。届いた場合は、日報フォームの報告者名や案件名が出勤レコードと一致しているか確認してください。

## 3. 承認フローのよくある質問

### Q3-1: 承認の流れを教えてください

**A:** 出退勤記録と日報でそれぞれ以下の承認フローになります。

**出退勤記録（T4）:**

```
[未承認] ---(班長が確認)---> [班長承認] ---(監督が確認)---> [監督承認]
    ^                           |                              |
    └------- [差戻し] <---------┘                              |
    ^                                                          |
    └------- [差戻し] <----------------------------------------┘
```

**日報（T5）:**

```
[未提出] --> [提出済] ---(班長が確認)---> [班長承認] ---(監督が確認)---> [監督承認]
                ^                           |                              |
                └------- [差戻し] <---------┘                              |
                ^                                                          |
                └------- [差戻し] <----------------------------------------┘
```

---

### Q3-2: 承認の通知はどこに届きますか？

**A:** Larkアプリの通知として届きます。

| タイミング | 通知先 | 自動化ルール |
|-----------|--------|------------|
| 日報が「提出済」になった時 | 班長 | A3: 承認依頼通知 |
| 日報が「班長承認」になった時 | 現場監督 | A4: 承認フロー進行 |
| 日報が「差戻し」になった時 | 報告者（作業員） | A5: 差戻し通知 |

---

### Q3-3: 差戻しされました。どうすればいいですか？

**A:** 以下の手順で対応してください。

1. Lark通知に記載された **承認コメント** を確認する
2. 指摘された箇所を特定する
3. Bitableのビューから該当レコードを開いて修正する
4. 修正後、承認状態を「提出済」に変更する

**よくある差戻し理由と対応:**

| 差戻し理由 | 対応方法 |
|-----------|---------|
| 作業時間が不正確 | 実際の作業時間に修正 |
| 作業内容が不明確 | 進捗メモに詳細を追記 |
| 写真が必要 | 写真を追加添付 |
| 安全事項の記載漏れ | 安全事項を記入 |

---

### Q3-4: 承認を取り消したい（班長・監督向け）

**A:** ビュー上で承認状態を前のステータスに変更してください。

| 現在の承認状態 | 変更先 |
|--------------|--------|
| 班長承認 | 提出済（日報）/ 未承認（出退勤） |
| 監督承認 | 班長承認 |

> **注意:** 承認を取り消すと、自動化ルールが再トリガーされる場合があります。例えば「提出済」に戻すと、A3（承認依頼通知）が再度班長に届く可能性があります。

---

### Q3-5: 承認操作はスマホからできますか？（班長・監督向け）

**A:** はい。Lark BiableアプリまたはLarkアプリ内のBitableビューから操作できます。

**手順:**

1. Larkアプリを開く
2. 該当するBitableを開く
3. 「未承認一覧」等の承認用ビューを表示
4. 該当レコードの「承認状態」セルをタップ
5. 「班長承認」または「監督承認」を選択
6. 変更は自動保存される

---

### Q3-6: 全員分をまとめて承認できますか？（班長・監督向け）

**A:** Bitable のビュー上で複数レコードを選択し、一括編集機能を使うことで可能です。

**手順:**

1. 承認用ビューを開く
2. 対象レコードをチェックボックスで複数選択
3. ツールバーの「一括編集」をクリック
4. 「承認状態」を「班長承認」等に変更
5. 保存

---

## 4. スマホ操作のトラブル

### Q4-1: フォームが開けません

**A:** 以下を順番に確認してください。

| 手順 | 確認内容 | 対処方法 |
|------|---------|---------|
| 1 | Larkアプリが最新版か | App Store / Google Playで更新 |
| 2 | インターネットに接続されているか | Wi-Fi / モバイルデータ通信を確認 |
| 3 | ブックマークのURLが正しいか | QRコードを再度読み取り |
| 4 | Larkにログインしているか | アカウント情報を確認してログイン |
| 5 | 上記すべてOKでも開けない | Larkアプリを一度終了して再起動 |
| 6 | それでもダメ | スマホを再起動 |

---

### Q4-2: Larkアプリにログインできません

**A:** 以下を確認してください。

1. **アカウント情報が正しいか**
   - 会社から渡されたメールアドレスとパスワードを確認
2. **パスワードを忘れた場合**
   - 「パスワードを忘れた場合」からリセット
   - または管理者に連絡してリセットしてもらう
3. **アカウントがロックされた場合**
   - 管理者に連絡してロック解除を依頼

---

### Q4-3: 通知が届きません

**A:** 以下を確認してください。

**Larkアプリ側の設定:**

1. Larkアプリを開く
2. 「設定」（歯車アイコン）をタップ
3. 「通知」をタップ
4. 通知が全てONになっているか確認

**スマホ側の設定:**

1. スマホの「設定」を開く
2. 「通知」を開く
3. 「Lark」を選択
4. 「通知を許可」がONか確認
5. 「バナー」「サウンド」「バッジ」がONか確認

**その他:**

- おやすみモード/集中モードがOFFか確認
- 省電力モードがONの場合、バックグラウンド通知が制限されることがあります

---

### Q4-4: フォーム送信でエラーが出ます

**A:** エラーの種類によって対処が異なります。

| エラーの状況 | 原因 | 対処方法 |
|------------|------|---------|
| 「必須項目が未入力です」 | *マークの項目が空 | 作業員名と案件を必ず選択 |
| 送信ボタンを押しても反応しない | 通信エラー | インターネット接続を確認して再試行 |
| 「送信に失敗しました」 | サーバーエラー | 数分待ってから再試行 |
| 画面が真っ白になる | アプリの不具合 | Larkアプリを終了して再起動 |

---

### Q4-5: フォームの選択リストに自分の名前がありません

**A:** 作業員マスタ（T2）に自分のレコードが登録されていないか、ステータスが「稼働中」になっていない可能性があります。管理者に連絡してください。

**管理者向け確認事項:**

1. 「T2: 作業員マスタ」に該当者のレコードがあるか確認
2. `fld_worker_status`（ステータス）が「稼働中」になっているか確認
3. フォームのフィルタ設定で「稼働中」のみ表示する設定になっているか確認

---

### Q4-6: フォームの選択リストに現場が出てきません

**A:** 案件マスタ（T1）に該当案件が登録されていないか、ステータスが「進行中」になっていない可能性があります。班長または管理者に連絡してください。

**管理者向け確認事項:**

1. 「T1: 案件マスタ」に該当案件のレコードがあるか確認
2. `fld_project_status`（ステータス）が「進行中」になっているか確認

---

### Q4-7: スマホが壊れた・紛失した

**A:** 班長に連絡してください。以下の対応が可能です。

1. **班長による代理入力:** 班長がBitableテーブルから直接出退勤レコード・日報を追加
2. **別端末からのアクセス:** PCやタブレットからLarkにログインしてフォームを利用
3. **新しいスマホの準備:** 管理者がLarkアカウントのパスワードリセットを実施し、新端末でログイン

## 5. データ・集計のトラブル

### Q5-1: 実働時間が0になっている

**A:** 以下の原因が考えられます。

| 原因 | 確認方法 | 対処 |
|------|---------|------|
| 出勤時刻が未入力 | `fld_clock_in`を確認 | 正しい出勤時刻を入力 |
| 退勤時刻が未入力 | `fld_clock_out`を確認 | 退勤打刻を実施するか、管理者が手動入力 |
| 退勤データが統合されていない | 出勤レコードに退勤時刻があるか確認 | 自動化A1の動作を確認（後述のQ6-1参照） |

**実働時間の計算式:**

```
実働時間 = (退勤時刻 - 出勤時刻) - 休憩時間

例: 出勤08:00, 退勤17:00, 休憩1.0h の場合
    (17:00 - 08:00) - 1.0 = 8.0時間
```

---

### Q5-2: 人工数が正しくありません

**A:** 人工数は実働時間から以下のルールで自動計算されます。

```
実働時間 7時間以上 --> 1.0 人工
実働時間 4時間以上 --> 0.5 人工
実働時間 4時間未満 --> 0   人工
```

**確認手順:**

1. まず実働時間（`fld_actual_hours`）が正しいか確認
2. 実働時間が正しければ、上記ルールに従って人工数も正しいはず
3. 実働時間が正しくない場合は、出勤時刻・退勤時刻・休憩時間を確認

> **注意:** 人工数の閾値（7時間/4時間）は企業ごとにカスタマイズ可能です。自社の閾値を管理者に確認してください。

---

### Q5-3: ダッシュボードにデータが表示されない

**A:** 以下を確認してください。

1. **フィルタ設定（期間）の確認**
   - ダッシュボードの日付フィルタが正しい期間に設定されているか
   - データが存在する期間を選択しているか
2. **ブラウザのリロード**
   - ブラウザの更新ボタンをクリック、またはF5キーを押す
3. **データの存在確認**
   - 元テーブル（T4/T5）に該当期間のレコードが存在するか確認
4. **権限の確認**
   - ダッシュボードの閲覧権限があるか管理者に確認

---

### Q5-4: 工数集計（T6）の金額が表示されない

**A:** 以下を確認してください。

1. **総人工数が入力されているか**
   - `fld_total_mandays`（総人工数）が空の場合、金額は計算されません
2. **作業員マスタの日当単価が設定されているか**
   - T2の`fld_daily_rate`（日当単価）が空の場合、金額は0になります

**金額の計算式:**

```
金額(円) = 総人工数 x 日当単価
```

---

### Q5-5: 休憩時間を変更したい

**A:** 退勤打刻フォーム（F2）で休憩時間を変更できます。デフォルトは1.0時間（昼休み1時間想定）です。

| 状況 | 休憩時間の設定 |
|------|-------------|
| 昼休み1時間のみ | 1.0（デフォルトのまま） |
| 昼休み+午後休憩あり | 1.5 |
| 休憩なし | 0 |
| 半日勤務で休憩なし | 0 |

> **注意:** 休憩時間を変更すると実働時間と人工数が再計算されます。

## 6. 自動化のトラブル

### Q6-1: 退勤打刻しても出勤レコードに退勤時刻が反映されない

**A:** 自動化A1（退勤データ統合）の動作に問題がある可能性があります。

**確認チェックリスト:**

| # | 確認事項 | 確認方法 |
|---|---------|---------|
| 1 | 出勤レコードの作業員名が退勤レコードと一致しているか | T4のフィルタで確認 |
| 2 | 出勤レコードの打刻日が退勤レコードと一致しているか | 日付形式（YYYY/MM/DD）を確認 |
| 3 | 出勤レコードの対象案件が退勤レコードと一致しているか | 案件名を確認 |
| 4 | 出勤レコードの打刻種別が「出勤」になっているか | fld_punch_typeを確認 |
| 5 | 自動化ルールA1が有効になっているか | 自動化設定画面でスイッチを確認 |
| 6 | 自動化ログにエラーがないか | 自動化の実行履歴を確認 |

**手動での対処:**

上記を確認しても解決しない場合、管理者が出勤レコードの退勤時刻（`fld_clock_out`）を直接編集してください。

> **補足:** 自動化は即時ではなく、レコード追加の数秒後に実行されます。退勤打刻後、5〜10秒程度待ってから確認してください。

---

### Q6-2: 日報リマインダー通知が来ない

**A:** 自動化A2の設定を確認してください。

| 確認事項 | 確認方法 |
|---------|---------|
| 自動化ルールA2が有効か | Bitable > 自動化 > A2のスイッチがON |
| 定時トリガーの時刻設定 | 毎日18:00に設定されているか |
| 通知先が正しく設定されているか | 作業員のLarkアカウントが正しいか |
| 当日の出勤レコードが存在するか | T4に当日の出勤レコードがあるか |
| Larkの通知設定がONか | スマホの通知設定を確認（Q4-3参照） |

---

### Q6-3: 承認通知が届かない

**A:** 自動化A3〜A5の設定を確認してください。

**共通の確認事項:**

1. **承認状態の値が正確か**
   - 全角・半角の違いに注意（「提出済」「提出済み」など表記揺れがないか）
   - 選択肢の値がテーブル定義と完全一致しているか
2. **通知先のLarkアカウントが正しいか**
   - T2（作業員マスタ）の`fld_lark_account`が正しく設定されているか
   - 班長の`fld_is_leader`がONになっているか
3. **自動化ルールが有効か**
   - 該当する自動化ルール（A3/A4/A5）のスイッチがONか

**ルールごとの確認:**

| ルール | トリガー条件 | 通知先 |
|--------|------------|--------|
| A3: 承認依頼通知 | 承認状態が「提出済」に変更 | 班長（fld_is_leader = true） |
| A4: 承認フロー進行 | 承認状態が「班長承認」に変更 | 現場監督（T1のfld_site_manager） |
| A5: 差戻し通知 | 承認状態が「差戻し」に変更 | 報告者 |

---

### Q6-4: 長時間勤務アラート（A6）が発動しない

**A:** 以下を確認してください。

1. **実働時間が12時間を超えているか**
   - アラート条件は `fld_actual_hours > 12` です（12時間ちょうどでは発動しません）
2. **退勤時刻が正しく記録されているか**
   - 退勤時刻が未入力の場合、実働時間は0のためアラートは発動しません
3. **数式フィールドの条件利用**
   - Bitableのバージョンによっては数式フィールドを自動化の条件に使えない場合があります
   - その場合、「長時間フラグ」チェックボックスフィールドを追加する方式Aを導入してください（05-automation-rules.md のA6セクション参照）

---

### Q6-5: GPS異常検知アラート（A7）が発動しない

**A:** 以下を確認してください。

1. **案件マスタにGPS座標が設定されているか**
   - T1の`fld_site_lat`（現場GPS緯度）と`fld_site_lng`（現場GPS経度）が入力されているか
2. **GPS許容半径が適切か**
   - `fld_gps_radius`のデフォルトは500m。広い現場では1000m以上に変更推奨
3. **出勤GPSが取得できているか**
   - GPS位置情報が取得できていない場合、距離計算ができずアラートは発動しません
4. **数式フィールド（fld_site_distance, fld_gps_alert）が正しく設定されているか**
   - ルックアップフィールドが正しく参照されているか確認

## 7. 管理者向けトラブルシューティング

### 7-1: 自動化ログの確認方法

自動化ルールが期待通りに動作しない場合、実行ログを確認してください。

**手順:**

1. Bitableを開く
2. 画面右上の「自動化」（稲妻アイコン）をクリック
3. 該当するルールをクリック
4. 「実行履歴」タブを開く
5. エラーがある場合は赤いアイコンで表示される
6. エラー詳細をクリックして内容を確認

**よくあるエラーと対処:**

| エラー内容 | 原因 | 対処 |
|-----------|------|------|
| 「検索結果が0件」 | 検索条件に一致するレコードがない | 検索条件（作業員名、日付、案件名）の一致を確認 |
| 「通知先が見つからない」 | Larkアカウントが未設定 | T2のfld_lark_accountを設定 |
| 「フィールド値が無効」 | データ型の不一致 | フィールドの型と入力値を確認 |
| 「実行回数上限に達した」 | プランの月間実行回数超過 | プランのアップグレードまたは不要なルールの無効化 |

---

### 7-2: 数式エラーの対処

**数式フィールドが「#ERROR」と表示される場合:**

1. **フィールド名の変更確認**
   - フィールド名が変更された場合、数式内の参照名も更新が必要
   - 例: フィールド名「実働時間」を「実働時間(h)」に変更した場合、数式内の参照も変更する

2. **参照先のデータ型確認**
   - 数値を期待する箇所にテキストが入っていないか確認
   - 日付フィールドと日付時刻フィールドの混在がないか確認

3. **数式のテスト方法**
   - テスト用レコードを作成し、各フィールドに値を入力して結果を確認
   - 段階的に値を変えて、どの入力で不正な結果になるか特定

**主要な数式一覧:**

| フィールド | 計算式 | テーブル |
|-----------|--------|---------|
| 実働時間 | `(退勤時刻 - 出勤時刻の分換算) / 60 - 休憩時間` | T4 |
| 人工数 | `IF(実働時間>=7, 1.0, IF(実働時間>=4, 0.5, 0))` | T4 |
| 金額 | `総人工数 x ルックアップ(日当単価)` | T6 |

---

### 7-3: データの一括修正

**ビューの一括編集機能を使う方法:**

1. 該当テーブルを開く
2. フィルタを設定して対象レコードを絞り込む
3. 対象レコードをチェックボックスで選択
4. ツールバーの「一括編集」をクリック
5. 変更するフィールドと値を指定
6. 実行

**CSVエクスポート → 修正 → 再インポートの方法（大量修正の場合）:**

1. テーブル右上の「...」メニューをクリック
2. 「CSVとしてエクスポート」を選択
3. CSVファイルをExcel等で開いて修正
4. テーブル右上の「...」メニューから「CSVからインポート」を選択
5. 修正したCSVファイルをアップロード

> **注意:** インポート時は「既存レコードを更新」オプションを選択し、主キー（自動番号フィールド）でマッチングしてください。「新規レコードとして追加」を選ぶとレコードが重複します。

---

### 7-4: バックアップ方法

月1回のCSVエクスポートを推奨します。

**エクスポート対象テーブルと推奨頻度:**

| テーブル | 推奨頻度 | 理由 |
|---------|---------|------|
| T1: 案件マスタ | 変更時 | レコード数が少ないため変更時でOK |
| T2: 作業員マスタ | 変更時 | 同上 |
| T3: 作業内容マスタ | 変更時 | 同上 |
| T4: 出退勤記録 | 月1回 | レコード数が多いため定期バックアップ |
| T5: 日報 | 月1回 | 同上 |
| T6: 工数集計 | 月1回 | 締め処理後にバックアップ |

**エクスポート手順:**

1. 該当テーブルを開く
2. テーブル右上の「...」メニューをクリック
3. 「CSVとしてエクスポート」を選択
4. ファイルをダウンロード
5. 「YYYYMM_テーブル名.csv」の形式でファイル名を変更
6. 所定のフォルダに保存

---

### 7-5: 新しい案件を追加する手順

1. 「T1: 案件マスタ」テーブルを開く
2. 「+ レコード追加」をクリック
3. 以下の項目を入力する
   - 案件名（必須）
   - 現場住所（必須）
   - 現場GPS緯度・経度（GPS異常検知を利用する場合）
   - GPS許容半径（デフォルト500m。広い現場は変更）
   - 工期開始日（必須）
   - 工期終了予定日
   - ステータス: 「準備中」または「進行中」
   - 現場監督: 担当者のLarkアカウントを選択
4. 保存

> **GPS座標の調べ方:** Googleマップで現場の中心点を長押しすると緯度・経度が表示されます。

---

### 7-6: 新しい作業員を追加する手順

1. 「T2: 作業員マスタ」テーブルを開く
2. 「+ レコード追加」をクリック
3. 以下の項目を入力する
   - 氏名（必須）
   - Larkアカウント（必須）: LarkのユーザーIDを選択
   - 職種（必須）
   - 所属班
   - 班長フラグ: 班長の場合はON
   - 日当単価（必須）
   - ステータス: 「稼働中」
   - 所属案件: 担当案件をリンク
4. 保存

---

### 7-7: 自動化ルールの導入・変更時の注意事項

1. **変更前にスクリーンショットを取得** -- 現在の設定を記録しておく
2. **テストレコードで動作確認** -- 本番データに影響がないことを確認
3. **連動するルールの影響を確認** -- 例: A1を変更するとA6（長時間アラート）に影響する可能性
4. **段階的に導入** -- 全ルールを一度に有効化せず、以下の順序を推奨

```
Phase 1 (1週目): A1（退勤データ統合）
Phase 2 (2週目): A3, A4, A5（承認フロー系）
Phase 3 (3週目): A2（日報リマインダー）
Phase 4 (4週目): A6, A7（アラート系）
```

## 9. 用語集

| 用語 | 説明 |
|------|------|
| Lark | ビジネスコミュニケーションアプリ。チャット、ビデオ通話、Bitableなどの機能を持つ |
| Bitable | Lark内のデータベース機能。テーブル、フォーム、ビュー、ダッシュボード、自動化が利用可能 |
| テーブル | データを管理する表。ExcelのシートのようなものがBitable上にある |
| フォーム | データを入力するための画面。スマホから簡単にデータを登録できる |
| ビュー | テーブルのデータをフィルタ・ソートして表示する画面 |
| レコード | テーブルの1行分のデータ。1件の出退勤記録や1件の日報がそれぞれ1レコード |
| フィールド | テーブルの1列（項目）。「作業員名」「出勤時刻」などがフィールド |
| 自動化（Automation） | 条件に応じて自動的に処理を実行する仕組み。通知送信やデータ転記などを自動化する |
| 人工（にんく） | 作業量の単位。1人が1日（7時間以上）働くと1.0人工 |
| 日当単価 | 1人工あたりの報酬金額 |
| 承認フロー | 提出されたデータを班長・現場監督が確認・承認する流れ |
| 差戻し | 承認者がデータに問題を見つけた場合に、提出者に修正を依頼する操作 |
| KY活動 | 危険予知活動。作業前に潜在的な危険を洗い出して対策を考える安全活動 |
| ヒヤリハット | 事故には至らなかったが、ヒヤリとした・ハッとした出来事。安全管理の重要な情報 |
| GPS | 全地球測位システム。スマホの位置情報を取得する仕組み |
| ルックアップ | 他のテーブルのデータを参照して表示する仕組み |
| リンクフィールド | テーブル間の関連を表すフィールド。別テーブルのレコードを参照できる |
