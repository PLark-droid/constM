# 管理者向けガイド

建設業向け勤怠・日報テンプレート -- Lark Bitable 管理者操作マニュアル

---

## はじめに

本ガイドは、建設業向け勤怠・日報管理システム（Lark Bitable）の **管理者** を対象としています。
管理者が行う設定変更、マスタデータの管理、自動化ルールの運用、ダッシュボードのカスタマイズ、権限管理、およびトラブルシューティングの手順を解説します。

### 管理者の役割

| 責務 | 内容 |
|------|------|
| マスタデータ管理 | 案件・作業員・作業内容の追加・更新・停止 |
| フォーム管理 | フォームURL発行、QRコード配布、フォーム設定変更 |
| 自動化ルール管理 | 自動化ルールの有効化・無効化・閾値変更 |
| ダッシュボード管理 | フィルタ設定、ウィジェット追加・変更 |
| 権限管理 | ロール別の共有設定・ビュー権限設定 |
| 月次業務 | 月次集計の作成、バックアップ、データクリーンアップ |
| トラブル対応 | エラー調査、データ修正、問い合わせ対応 |

### 前提条件

- Lark Bitableの管理者権限を持っていること
- Larkの管理コンソールへのアクセス権を持っていること（ユーザー管理が必要な場合）

## 1. マスタデータの管理

### 1-1. 案件マスタの管理（T1: `tbl_projects`）

案件マスタは現場・案件の基本情報を管理するテーブルです。管理者が作成・更新を行います。

#### 新規案件の追加手順

```
1. Bitable を開き、「T1: 案件マスタ」テーブルを選択する
2. 「全案件一覧」ビューを開く
3. テーブル最下部の「+」（新規レコード追加）をクリックする
4. 以下のフィールドを入力する:

   [必須フィールド]
   - 案件名       : 現場の名称（例: ○○ビル新築工事）
   - 現場住所     : 現場の住所
   - 工期開始     : 工事開始日（YYYY/MM/DD）
   - ステータス   : 「準備中」を選択（デフォルト）

   [推奨フィールド]
   - 発注者       : 元請け・発注者名
   - 工期終了     : 工事終了予定日
   - 現場監督     : 担当監督のLarkアカウントを選択
   - 現場GPS緯度  : 現場中心点の緯度（小数点以下6桁）
   - 現場GPS経度  : 現場中心点の経度（小数点以下6桁）
   - GPS許容半径  : GPS打刻の許容距離（メートル単位、デフォルト500）

5. 入力が完了したら、レコード外の領域をクリックして保存する
```

#### 案件ステータスの変更

案件の進捗に応じて、ステータスを変更します。

| 遷移 | タイミング | 操作 |
|------|----------|------|
| 準備中 → 進行中 | 工事が開始されたとき | ステータスセルをクリックし「進行中」を選択 |
| 進行中 → 完了 | 工事が完了・引渡しが終わったとき | ステータスセルをクリックし「完了」を選択 |
| 進行中 → 中止 | 工事が中止になったとき | ステータスセルをクリックし「中止」を選択 |

**注意:** ステータスを「完了」や「中止」にすると、以下に影響します。

- D3（案件別詳細ダッシュボード）の案件選択フィルタに表示されなくなる（進行中のみ表示の設定の場合）
- V1-2（進行中案件ビュー）のカンバンから消える

完了した案件の出退勤データは引き続きテーブルに残るため、月次集計やダッシュボードの過去データ参照には影響しません。

#### GPS座標の設定方法（Googleマップから取得）

GPS座標を設定すると、出退勤打刻時のGPS異常検知（自動化ルールA7）が有効になります。

```
手順:

1. パソコンまたはスマートフォンで Google マップ（maps.google.com）を開く
2. 現場の住所を検索する
3. 現場の中心付近を地図上で右クリック（スマホの場合は長押し）する
4. 表示された座標をタップまたはクリックしてコピーする

   例: 35.681236, 139.767125
        ^^^^^^^^  ^^^^^^^^^^
        緯度      経度

5. Bitable の案件レコードを開く
6. 「現場GPS緯度」に緯度の値を入力する（例: 35.681236）
7. 「現場GPS経度」に経度の値を入力する（例: 139.767125）
8. 「GPS許容半径(m)」を設定する（デフォルト: 500）
```

#### GPS許容半径の調整ガイドライン

現場の規模に応じて、GPS許容半径を調整してください。半径が小さすぎると正当な打刻でもGPS異常として検知され、大きすぎると不正打刻を検知できません。

| 現場の規模 | 例 | 推奨半径 | 備考 |
|-----------|-----|---------|------|
| 小規模 | 戸建て住宅、小規模リフォーム | 200-300m | GPS精度のばらつきを考慮 |
| 中規模 | マンション、中規模ビル | 500m | デフォルト値。多くの現場で適切 |
| 大規模 | 工業団地、大規模開発 | 1000m以上 | 敷地が広い場合は大きめに設定 |
| 屋内・地下 | 地下工事、屋内改修 | 1000m以上 | GPS信号が弱くなるため広めに設定 |

**補足:** GPS座標が未設定の案件ではGPS異常検知（A7）は動作しません。GPS検知が不要な案件では空欄のままで問題ありません。

### 1-2. 作業員マスタの管理（T2: `tbl_workers`）

作業員マスタは全作業員の基本情報を管理するテーブルです。

#### 新規作業員の登録

```
1. 「T2: 作業員マスタ」テーブルの「全作業員一覧」ビューを開く
2. テーブル最下部の「+」をクリックして新規レコードを作成する
3. 以下のフィールドを入力する:

   [必須フィールド]
   - 氏名           : フルネーム（例: 山田太郎）
   - Larkアカウント  : 該当作業員のLarkユーザーを選択
   - 職種           : 主な職種を選択
                      （とび / 鉄筋工 / 型枠大工 / 左官 / 電気工 /
                        配管工 / 塗装工 / 解体工 / 重機オペレーター / その他）
   - 日当単価(円)    : 1人工あたりの日当金額（税抜）
   - ステータス      : 「稼働中」を選択（デフォルト）

   [推奨フィールド]
   - 所属班         : 班名（例: A班、鈴木組）
   - 班長フラグ     : 班長の場合はチェックをONにする
   - 連絡先         : 緊急連絡先の電話番号
   - 所属案件       : 担当する案件をリンクで選択（複数選択可）

4. 入力完了後、レコード外をクリックして保存する
```

#### Larkアカウントとの紐付け

作業員がLark通知を受け取るためには、`fld_lark_account`（Larkアカウント）フィールドに正しいユーザーが紐づいている必要があります。

```
確認手順:

1. 「全作業員一覧」ビューで対象作業員のレコードを開く
2. 「Larkアカウント」フィールドをクリックする
3. Lark組織内のユーザー一覧が表示される
4. 作業員名で検索し、正しいアカウントを選択する

※ アカウントが見つからない場合:
   - Lark管理コンソールでユーザーが作成済みか確認する
   - 作業員がLarkアプリにログイン済みか確認する
```

#### 案件への作業員アサイン

```
1. 対象作業員のレコードを開く
2. 「所属案件」フィールドをクリックする
3. 案件マスタ（T1）のレコード一覧が表示される
4. 担当する案件を選択する（複数選択可）
5. 案件の変更（追加・削除）があった場合は、同じ操作でリンクを更新する
```

**補足:** 1人の作業員が複数案件に所属できます。出退勤打刻時のフォームでは、所属案件に関係なく全ての進行中案件が選択肢として表示されます。

#### 退職時の処理

作業員が退職した場合、レコードを削除するのではなく、ステータスを変更します。過去の勤怠データとの紐づきが維持されます。

```
1. 対象作業員のレコードを開く
2. 「ステータス」を「退職済」に変更する
3. 「所属案件」のリンクを全て外す（任意）
4. 「備考」（ない場合はフィールドを追加）に退職日を記録する（推奨）

   例: 2026/03/31 退職

※ ステータスが「退職済」の作業員は:
   - 出勤打刻・日報提出フォームの選択リストに表示されなくなる
     （ビューのフィルタ条件で「稼働中」のみ表示の場合）
   - 過去データは引き続き参照可能
```

#### 日当単価の変更

日当単価を変更する場合は、変更履歴を追跡できるよう記録を残してください。

```
1. 対象作業員のレコードを開く
2. 「日当単価(円)」の値を新しい金額に変更する
3. 「備考」に変更日と旧単価を記録する（推奨）

   例: 2026/04/01 日当 18,000円 → 20,000円 に変更

※ 注意:
   - 単価変更は変更後の工数集計（T6）に影響する
   - 変更前に作成済みのT6レコードの金額は自動更新されない
   - 過去月分の金額を遡及修正する場合はT6レコードを手動で再計算する
```

### 1-3. 作業内容マスタの管理（T3: `tbl_work_types`）

作業内容マスタは日報で選択する作業分類を管理するテーブルです。初期設定後は頻繁な変更は発生しません。

#### 作業項目の追加

```
1. 「T3: 作業内容マスタ」テーブルの「作業内容一覧」ビューを開く
2. テーブル最下部の「+」をクリックして新規レコードを作成する
3. 以下のフィールドを入力する:

   - 作業大分類   : 工種の大分類を選択
                    （仮設 / 土工 / 基礎 / 躯体 / 仕上 / 設備 / 外構 / その他）
   - 作業名       : 具体的な作業名（例: 鉄筋組立、型枠建込）
   - 標準人工     : 標準的な所要人工（参考値。空欄可）

4. 保存後、日報提出フォーム（F3）の選択肢に自動的に追加される
```

#### 作業大分類の使い分け

| 大分類 | 含まれる作業例 |
|--------|-------------|
| 仮設 | 足場組立、仮囲い、安全施設 |
| 土工 | 掘削、埋戻し、盛土、地盤改良 |
| 基礎 | 杭打ち、基礎配筋、基礎コンクリート |
| 躯体 | 鉄筋組立、型枠建込、コンクリート打設 |
| 仕上 | 左官、タイル、塗装、内装 |
| 設備 | 電気配線、配管、空調、給排水 |
| 外構 | 舗装、植栽、フェンス |
| その他 | 清掃、搬入出、準備作業、打合せ |

#### 不要項目の非表示化

使わなくなった作業項目は、削除ではなく非表示にすることを推奨します。削除すると過去の日報データとのリンクが切れてしまいます。

```
方法: ビューのフィルタで非表示にする

1. T3の「作業内容一覧」ビューを開く
2. T3に「有効フラグ」チェックボックスフィールドを追加する（初回のみ）
3. 全ての既存レコードの「有効フラグ」をONにする
4. 非表示にしたい項目の「有効フラグ」をOFFにする
5. ビューのフィルタに「有効フラグ = ON」を追加する

※ 日報フォームのリンク選択にもフィルタが反映され、
   OFFの項目はフォーム上で選択できなくなります。
   （ビューの設定に依存するため、動作確認を行ってください）
```

## 2. フォームの管理

本システムでは3つのフォームを使用します。

| フォーム | 対象テーブル | 用途 |
|---------|------------|------|
| F1: 出勤打刻 | T4: 出退勤記録 | 毎朝の出勤打刻 |
| F2: 退勤打刻 | T4: 出退勤記録 | 毎夕の退勤打刻 |
| F3: 日報提出 | T5: 日報 | 毎夕の日報提出 |

### 2-1. フォームURLの管理

#### フォームURLの確認・取得

```
1. Bitable を開き、対象テーブルを選択する
2. ツールバーの「フォーム」をクリックする
3. 既存フォームの一覧が表示される
4. 対象フォームの「共有」ボタンをクリックする
5. 共有リンク（URL）が表示される
6. 「コピー」をクリックしてURLを取得する
```

#### QRコードの発行

```
1. フォームの共有URLを取得する（上記手順）
2. QRコード生成サービスを使用する
   （例: qr.io、Googleの QR Code Generator など）
3. URLを入力してQRコードを生成する
4. QRコード画像をダウンロードする
5. 印刷して現場事務所に掲示する

推奨: 以下の3つのQRコードをまとめて1枚のシートに印刷する
   - 出勤打刻（F1）
   - 退勤打刻（F2）
   - 日報提出（F3）
```

#### フォームの一時停止（工事完了時など）

工事が完了した案件のフォームを無効化する場合:

```
方法A: フォームの共有リンクを無効化する
1. 対象フォームの共有設定を開く
2. 共有リンクをOFFにする
3. 既存のURLは無効になり、アクセスできなくなる

方法B: 案件ステータスを「完了」にする
1. 案件マスタ（T1）で対象案件のステータスを「完了」に変更する
2. フォーム自体はアクティブのまま残すが、
   作業員が案件を選択しても実運用上は使用されない
3. 出退勤データが誤って入力された場合は手動で削除する

※ 方法Aは全てのフォームを一括停止したい場合に有効です。
   案件ごとの制御はフォーム単位では行えないため、
   運用ルールで対応します。
```

#### フォームのアクセス権限設定

```
1. フォームの共有設定を開く
2. 「公開範囲」を設定する:
   - 「組織内メンバーのみ」: Lark組織に所属するメンバーのみアクセス可（推奨）
   - 「リンクを知っている全員」: URLを知っていれば誰でもアクセス可
   - 「特定のメンバー」: 指定したユーザーのみアクセス可

推奨設定: 「組織内メンバーのみ」
```

### 2-2. フォームのカスタマイズ

#### フィールドの追加・非表示

```
フィールドを非表示にする:
1. フォーム編集画面を開く
2. 非表示にしたいフィールドの右側にある目のアイコンをクリックする
3. フィールドがフォーム上に表示されなくなる
   （テーブルのフィールド自体は残ります）

フィールドを表示に戻す:
1. フォーム編集画面で「フィールドを追加」をクリックする
2. 非表示になっているフィールドの一覧から選択する
3. ドラッグ&ドロップで表示位置を調整する
```

#### 必須項目の変更

```
1. フォーム編集画面を開く
2. 対象フィールドをクリックする
3. フィールド設定パネルが表示される
4. 「必須」のトグルをON/OFFにする
5. 保存する

注意:
- 以下のフィールドは必須のまま変更しないでください
  （データ整合性に影響します）:
  - 作業員（fld_att_worker / fld_reporter）
  - 対象案件（fld_att_project / fld_report_project）
  - 打刻日 / 報告日
  - 打刻種別（固定値フィールド）
  - 承認状態（固定値フィールド）
```

#### デフォルト値の変更

```
1. フォーム編集画面を開く
2. 対象フィールドをクリックする
3. フィールド設定パネルで「デフォルト値」を変更する

例: 休憩時間のデフォルトを変更する場合
   - 現在: 1.0（昼休み1時間を想定）
   - 変更後: 0.5（半日勤務の現場で使用する場合）
```

## 3. 自動化ルールの管理

本システムでは7つの自動化ルールが設定されています。

| ID | ルール名 | 目的 | 優先度 |
|----|---------|------|--------|
| A1 | 退勤データ統合 | 退勤時刻を出勤レコードに転記 | 最高 |
| A2 | 日報未提出リマインダー | 毎日18:00に全作業員へリマインド通知（フラグ方式で未提出者限定も可） | 高 |
| A3 | 承認依頼通知 | 日報提出時に班長に通知 | 高 |
| A4 | 承認フロー進行 | 班長承認後に監督に通知 | 高 |
| A5 | 差戻し通知 | 差戻し時に報告者に通知 | 高 |
| A6 | 長時間勤務アラート | 実働12時間超で警告通知 | 最高 |
| A7 | GPS異常検知 | GPS許容範囲外の打刻を通知 | 中 |

### 3-1. 自動化の動作確認

#### 自動化ログの確認方法

```
1. Bitable を開く
2. 画面右上の「自動化」（稲妻アイコン）をクリックする
3. 自動化ルールの一覧が表示される
4. 確認したいルールをクリックする
5. 「実行ログ」タブをクリックする
6. 過去の実行履歴が表示される:
   - 実行日時
   - 実行結果（成功 / 失敗）
   - トリガーレコードの情報
   - エラーメッセージ（失敗の場合）
```

#### エラー発生時の対処

| エラー種別 | 考えられる原因 | 対処法 |
|-----------|-------------|--------|
| 検索結果が0件 | 検索条件に一致するレコードがない | 検索条件を確認。日付フォーマットの不一致がないか確認する |
| 通知送信失敗 | 通知先のLarkアカウントが無効 | T2の `fld_lark_account` が正しく設定されているか確認する |
| フィールド更新失敗 | 対象フィールドのデータ型不一致 | フォーム側のフィールド型と自動化の設定を照合する |
| ルール自体が動かない | ルールが無効（OFF）になっている | ルールのスイッチがONか確認する |
| 実行回数上限到達 | 月間実行回数がプラン上限を超過 | Lark管理コンソールで残り回数を確認。プランのアップグレードを検討する |

#### ルールの手動テスト

```
1. 自動化ルール一覧から対象ルールをクリックする
2. ルール編集画面の「テスト実行」ボタンをクリックする
   （Lark Bitableのバージョンにより利用可否が異なる）
3. テスト用のトリガーレコードを指定する
4. 実行結果を確認する

テスト実行ボタンがない場合:
- テストデータを手動でテーブルに追加し、ルールが発動するか確認する
- テスト後、テストデータは削除すること
```

### 3-2. 通知設定のカスタマイズ

#### リマインダー時刻の変更（A2: 日報未提出リマインダー）

デフォルトでは毎日18:00に日報未提出リマインダーが送信されます。

```
変更手順:

1. 自動化ルール一覧から「A2: 日報未提出リマインダー」をクリックする
2. トリガー設定の「定時実行」を編集する
3. 「実行時刻」を変更する
   例: 18:00 → 17:30（退勤が早い現場の場合）
   例: 18:00 → 19:00（残業が多い現場の場合）
4. 保存する

注意:
- 時刻を早くしすぎると、まだ作業中の作業員にも通知が届きます
- 時刻を遅くしすぎると、日報提出の催促効果が薄れます
- 推奨: 退勤予定時刻の30分〜1時間後
```

#### 通知先の変更

各自動化ルールの通知先は以下の方法で変更できます。

```
1. 対象ルールを開く
2. 「通知を送信」アクションを編集する
3. 「通知先」フィールドの設定を変更する:

   方法A: 動的通知先（検索結果から）
   - 検索アクションの結果から Larkアカウント を参照する

   方法B: 固定通知先
   - 特定のLarkユーザーまたはLarkグループを直接指定する
   - 複数の管理者に常に通知したい場合に有効
```

#### 通知メッセージの編集

```
1. 対象ルールを開く
2. 「通知を送信」アクションを編集する
3. 「通知内容」のテキストを変更する

   変数の挿入:
   - {フィールド名} の形式でトリガーレコードの値を埋め込める
   - 例: {fld_att_worker} → 作業員名が表示される
   - 例: {fld_att_date} → 打刻日が表示される

4. 保存する
```

### 3-3. 長時間勤務アラートの閾値変更（A6）

デフォルトでは実働時間が **12時間** を超えた場合にアラートが発動します。

```
閾値変更手順:

1. 自動化ルール「A6: 長時間勤務アラート」を開く
2. トリガーの条件を編集する
3. 「fld_actual_hours > 12」の「12」を変更する
   例: 10（より厳しい基準にする場合）
   例: 14（基準を緩和する場合）
4. 保存する

※ 数式フィールドを条件に使用できない場合:
   T4に「長時間フラグ」（fld_long_hours_flag）チェックボックスを
   追加している場合は、そのフラグの計算式も合わせて変更する。

   数式例（閾値を10時間に変更）:
   IF(fld_actual_hours > 10, true, false)
```

## 4. ダッシュボードの管理

### 4-1. フィルタの設定変更

#### 期間フィルタの調整

各ダッシュボードには期間フィルタ（F1）が設定されています。

```
フィルタのデフォルト値を変更する:

1. ダッシュボードを開く
2. 上部の「フィルタ」パネルを展開する
3. 期間フィルタの設定を変更する:

   D1（現場日次）:
   - デフォルト: 本日
   - 変更例: 直近3日間（前日の見落としを確認したい場合）

   D2（月次管理）:
   - デフォルト: 今月
   - 変更例: 今月 + 先月（比較分析したい場合）

   D3（案件別詳細）:
   - デフォルト: 全期間
   - 変更例: 直近3ヶ月（長期案件でデータ量が多い場合）

4. プリセット値:
   本日 / 今週 / 今月 / 先月 / カスタム範囲
```

#### 案件フィルタの設定（D3用）

```
D3（案件別詳細ダッシュボード）の案件選択フィルタを調整する:

2. 案件選択フィルタをクリックする

   デフォルト: 「進行中」の案件のみ表示

   - 「進行中」のみ → fld_project_status = "進行中"

6. 「追加」をクリックする

3. 新しい権限レベルを選択する

## 6. カスタマイズガイド

### 6-1. 人工数計算のカスタマイズ

T4（出退勤記録）の人工数は以下のデフォルト閾値で計算されています。

| 実働時間 | 人工数 |
|---------|--------|
| 7時間以上 | 1.0人工 |
| 4時間以上7時間未満 | 0.5人工 |
| 4時間未満 | 0人工 |

#### 閾値の変更方法

```
1. T4（出退勤記録）テーブルを開く
2. 「人工数」フィールド（fld_manday）のヘッダーをクリックする
3. 「フィールドを編集」を選択する
4. 数式を変更する:

   デフォルトの数式:
   IF(fld_actual_hours >= 7, 1.0, IF(fld_actual_hours >= 4, 0.5, 0))

   変更例（6時間を1.0人工、3時間を0.5人工にする場合）:
   IF(fld_actual_hours >= 6, 1.0, IF(fld_actual_hours >= 3, 0.5, 0))

   変更例（0.25人工の区分を追加する場合）:
   IF(fld_actual_hours >= 7, 1.0,
     IF(fld_actual_hours >= 4, 0.5,
       IF(fld_actual_hours >= 2, 0.25, 0)))

5. 「保存」をクリックする
6. 既存レコードの人工数が自動的に再計算される
```

#### テスト方法

```
1. T4にテスト用の出勤レコードを作成する
2. 出勤時刻と退勤時刻を設定して、様々な実働時間のパターンを確認する:
   - 閾値ちょうど（例: 実働7.0h → 1.0人工 になるか）
   - 閾値未満（例: 実働6.9h → 0.5人工 になるか）
   - 閾値境界（例: 実働4.0h → 0.5人工 になるか）
   - 最小値（例: 実働0h → 0人工 になるか）
3. 確認後、テストレコードを削除する
```

### 6-2. 承認フローのカスタマイズ

#### 2段階承認から1段階承認への変更

デフォルトの承認フローは2段階（班長承認 → 監督承認）です。小規模な組織では1段階（監督のみ）に簡略化できます。

```
変更手順:

1. T4の「承認状態」フィールドの選択肢を変更する:
   - 変更前: 未承認 / 班長承認 / 監督承認 / 差戻し
   - 変更後: 未承認 / 承認済 / 差戻し

2. T5の「承認状態」フィールドの選択肢を変更する:
   - 変更前: 未提出 / 提出済 / 班長承認 / 監督承認 / 差戻し
   - 変更後: 未提出 / 提出済 / 承認済 / 差戻し

3. 自動化ルールを調整する:
   - A3（承認依頼通知）: 通知先を監督に変更
   - A4（承認フロー進行）: 無効化する（1段階のため不要）

4. ビューのフィルタを調整する:
   - 「未承認一覧」ビュー: 承認状態 = "未承認" のまま
   - 「班長承認済一覧」ビュー: 不要になるため削除または非表示

5. ダッシュボードの条件付き書式を調整する:
   - 「班長承認」の条件を「承認済」に変更
```

#### 承認状態の選択肢変更

```
1. テーブルを開く（T4 または T5）
2. 「承認状態」フィールドのヘッダーをクリックする
3. 「フィールドを編集」を選択する
4. 「単一選択」の選択肢を追加・削除・名称変更する

注意:
- 選択肢を削除すると、その値を持つ既存レコードの値が空になります
- 選択肢の名称を変更すると、既存レコードの値も自動的に更新されます
- 自動化ルールの条件に選択肢名を使用している場合は、
  ルール側も合わせて修正してください
```

### 6-3. フィールドの追加

テーブルに新しいフィールドを追加する一般的な手順です。

```
1. 対象テーブルを開く
2. テーブルの右端にある「+」（フィールド追加）をクリックする
3. フィールド名を入力する
4. フィールドの型を選択する:
   - テキスト: 自由入力のテキスト
   - 数値: 数字（小数点設定可能）
   - 単一選択: 選択肢から1つ選ぶ
   - 複数選択: 選択肢から複数選ぶ
   - 日付: 日付・日時
   - チェックボックス: ON/OFF
   - 数式: 計算式による自動算出
   - リンク: 他テーブルのレコードへのリンク
   - ルックアップ: リンク先のフィールド値を参照
   - 通貨: 金額（通貨記号・カンマ区切り設定可能）

5. フィールド固有の設定を行う
6. 「保存」をクリックする
```

```

## 8. 月次業務

### 8-1. 月次集計の作成

T6（工数集計）テーブルに月次の集計レコードを作成します。ダッシュボードD2（月次管理）の補助データとして使用します。

```
作成手順:

1. 月末の最終営業日に実施する
2. 全ての出退勤レコード（T4）が承認済みであることを確認する
   - T4の「未承認一覧」ビューを開き、当月のレコードが0件であることを確認
   - 未承認レコードがある場合は、先に承認処理を完了する

3. T4の「個人別月次」ビューを開く
   - 作業員ごとにグループ化されたサマリ行を確認する
   - 各作業員の出勤日数・実働時間合計・人工数合計を確認する

4. T6テーブルの「月次集計一覧」ビューを開く

5. 作業員 x 案件の組み合わせごとにレコードを作成する:
   - 対象年月: 当月の1日を入力（例: 2026/03/01）
   - 作業員: 対象の作業員をリンクで選択
   - 対象案件: 対象の案件をリンクで選択
   - 出勤日数: T4のデータから集計した値を入力
   - 総実働時間(h): T4のデータから集計した値を入力
   - 総人工数: T4のデータから集計した値を入力
   - 金額(円): 自動計算（総人工数 x 日当単価）

6. 全ての組み合わせの入力が完了したら、
   ダッシュボードD2で金額合計を確認する
```

#### T4との照合

```
照合手順:

1. T4の「個人別月次」ビューでグループ集計を確認する
2. T6の当月レコードの値と照合する
3. 差異がある場合は:
   - T4に未承認のレコードがないか確認する
   - T4に「退勤」レコードのみ（出勤レコードなし）のものがないか確認する
   - 休憩時間が異常な値になっていないか確認する
```

### 8-2. データクリーンアップ

#### テストデータの削除

```
1. T4、T5のテーブルを開く
2. フィルタで以下の条件に該当するレコードを検索する:
   - 作業員名が「テスト」「test」などのテスト用レコード
   - 打刻日/報告日が不自然に古い、または未来のレコード
   - 備考欄に「テスト」と記載されているレコード
3. 該当レコードを選択して削除する

注意:
- 削除前にレコードの内容を確認し、誤って実データを削除しないようにする
- 削除は取り消しできないため、不安な場合は先にエクスポートしておく
```

#### 不要レコードの整理

```
以下のレコードを定期的に確認・整理してください:

[1] 退勤レコード（打刻種別=退勤）
    → 自動化A1で出勤レコードに退勤時刻が転記された後、
      退勤レコード自体はログとして残る設計
    → 不要と判断した場合はまとめて削除しても可
      （出勤レコード側に退勤時刻が転記済みであることを確認の上）

[2] 重複レコード
    → 同一日・同一作業員・同一案件で複数の出勤レコードがある場合
    → 正しい1件を残し、不要なレコードを削除する

[3] 差戻し後に放置されたレコード
    → 承認状態が「差戻し」のまま長期間放置されているレコード
    → 報告者に修正・再提出を促すか、管理者が修正する
```

## 9. トラブルシューティング

### 自動化が動かない

| 確認項目 | 手順 |
|---------|------|
| ルールがONか確認 | 自動化一覧でルールのスイッチがONであることを確認する |
| 実行ログを確認 | ルールの実行ログタブでエラーが出ていないか確認する |
| 条件を確認 | トリガー条件が正しいか確認する（フィールド名の変更があると条件が壊れる場合がある） |
| 月間実行回数を確認 | プランの月間上限に達していないか確認する |
| フィールド型の一致を確認 | 自動化の条件やアクションで参照しているフィールドの型が変更されていないか確認する |

```
復旧手順:

1. 自動化ルールの実行ログでエラー内容を確認する
2. エラーの原因に応じて修正する:
   - 「レコードが見つからない」→ 検索条件の見直し
   - 「フィールドが見つからない」→ フィールド名/IDの確認
   - 「権限エラー」→ 自動化のオーナーの権限を確認
3. ルールを一度OFFにして、再度ONにする
4. テストデータで動作確認を行う
```

### 数式エラー

| エラー内容 | 原因 | 対処 |
|-----------|------|------|
| #ERROR! | 数式の構文エラー | 数式エディタで括弧の対応、関数名のスペルを確認する |
| #REF! | 参照先のフィールドが削除された | フィールド名を確認し、正しいフィールドに修正する |
| 0が表示される | 条件が満たされていない | IF文の条件と元データ（出勤時刻・退勤時刻）を確認する |
| 空欄が表示される | 入力データが不足している | 必要なフィールド（出勤時刻・退勤時刻など）が入力されているか確認する |

```
実働時間が計算されない場合の確認:

1. 出勤時刻（fld_clock_in）が入力されているか確認する
2. 退勤時刻（fld_clock_out）が入力されているか確認する
   → 退勤時刻は自動化A1で転記されるため、A1が正しく動作しているか確認
3. 休憩時間（fld_break_hours）が異常な値になっていないか確認する
4. 数式フィールドの式が変更されていないか確認する:
   IF(AND(fld_clock_in, fld_clock_out),
     MAX(
       (HOUR(fld_clock_out)*60 + MINUTE(fld_clock_out)
        - HOUR(fld_clock_in)*60 - MINUTE(fld_clock_in)) / 60
       - fld_break_hours,
       0
     ),
     0
   )
```

### 権限エラー

| 症状 | 原因 | 対処 |
|------|------|------|
| テーブルが見えない | Bitableの共有設定にユーザーが含まれていない | 「共有」設定でユーザーを追加する |
| 編集できない | 閲覧者権限になっている | 権限を「編集者」に変更する |
| フォームが送信できない | フォームのアクセス権限が制限されている | フォームの共有設定を確認する |
| ダッシュボードが見えない | ダッシュボードが共有されていない | ダッシュボードの共有設定を確認する |

```
確認手順:

1. Bitableの「共有」設定を開く
2. 問題が発生しているユーザーの権限レベルを確認する
3. 必要に応じて権限を変更する
4. ユーザーにBitableのURLを再共有する
   （URLの期限切れや変更の可能性がある）
```

### フォームが表示されない

| 症状 | 原因 | 対処 |
|------|------|------|
| 「このフォームは存在しません」 | フォームが削除された | フォームを再作成し、新しいURLを配布する |
| 「アクセス権限がありません」 | 公開範囲の設定が不適切 | フォーム共有設定で「組織内メンバーのみ」を確認する |
| ページが読み込めない | URLが間違っている | 正しいフォームURLを再取得して配布する |
| 選択肢が表示されない | リンク先テーブルのデータが空 | マスタテーブルにデータが登録されているか確認する |

```
フォームURL再発行手順:

1. Bitableを開く
2. 対象テーブルの「フォーム」をクリックする
3. 対象フォームを選択する
4. 「共有」をクリックする
5. 共有リンクを「ON」にする
6. URLをコピーする
7. 新しいQRコードを生成する
8. 現場事務所のQRコードを差し替える
9. Larkグループチャットでも新しいリンクを周知する
```

各フェーズで1週間程度の運用テストを行い、問題がないことを確認してから次のフェーズに進んでください。
