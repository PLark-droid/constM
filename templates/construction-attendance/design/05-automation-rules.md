# 自動化ルール設計書（Automation Rules）

建設業向け勤怠・日報テンプレート — Lark Bitable 自動化ルール定義

---

## 概要

Lark Bitableの自動化（Automation）機能を使い、データ統合・通知・アラートを自動化する。
本設計書では7つの自動化ルールを定義し、手作業を最小化しつつ現場運用を円滑に回す仕組みを構築する。

### 自動化の目的

| 目的 | 対応ルール |
|------|-----------|
| データ統合（出退勤の出勤・退勤レコード統合） | A1 |
| リマインダー（未提出・未対応の検知と通知） | A2 |
| 承認フロー推進（ステータス変更に連動した通知） | A3, A4, A5 |
| 安全管理アラート（労務・位置の異常検知） | A6, A7 |

### 前提条件

- Lark Bitableの自動化機能（Automation）が利用可能なプランであること
- 各作業員がLarkアカウントを保有し、T2（作業員マスタ）の`fld_lark_account`に登録済みであること
- 自動化ルールの作成にはBitableの管理者権限が必要

## A1: 退勤データ統合

### 基本情報

| 項目 | 内容 |
|------|------|
| ルール名 | 退勤データ統合 |
| 目的 | 退勤フォームで作成されたレコードの退勤時刻を、同一日の出勤レコードに転記する |
| トリガー | T4（出退勤記録）にレコードが追加された時 |
| トリガー条件 | 打刻種別 = 「退勤」 |
| 優先度 | 最高（データ整合性の根幹） |

### アクション詳細

| ステップ | アクション | 対象 |
|---------|-----------|------|
| 1 | レコード検索 | T4を検索: 作業員=同一 AND 打刻日=同一 AND 対象案件=同一 AND 打刻種別=「出勤」 |
| 2 | フィールド転記（退勤時刻） | 該当レコードの`fld_clock_out`に、新レコードの退勤時刻を転記 |
| 3 | フィールド転記（退勤GPS） | 該当レコードの`fld_gps_out`に、新レコードの退勤GPS座標を転記 |

### エラー時の挙動

- 出勤レコードが見つからない場合は転記せず、退勤レコードのみ残る
- 該当する出勤レコードが複数存在した場合は、最新（作成日時が最も新しい）レコードに転記される

### 設定手順（Bitable UIでの操作ステップ）

```
1. Bitableを開き、画面右上の「自動化」（稲妻アイコン）をクリック
2. 「+ ルールを作成」をクリック
3. ルール名に「A1: 退勤データ統合」と入力

【トリガー設定】
4. トリガーの種類で「レコードが追加された時」を選択
5. テーブルで「T4: 出退勤記録（tbl_attendance）」を選択
6. 「条件を追加」をクリック
7. 条件: fld_punch_type（打刻種別） = 「退勤」 を設定

【アクション1: レコード検索】
8. 「+ アクションを追加」をクリック
9. アクションの種類で「レコードを検索」を選択
10. 検索先テーブル: 「T4: 出退勤記録（tbl_attendance）」を選択
11. 検索条件を以下のように設定:
    - fld_att_worker（作業員） = トリガーレコードの fld_att_worker（作業員）
    - fld_att_date（打刻日） = トリガーレコードの fld_att_date（打刻日）
    - fld_att_project（対象案件） = トリガーレコードの fld_att_project（対象案件）
    - fld_punch_type（打刻種別） = 「出勤」
12. 検索結果の上限: 1件に設定

【アクション2: レコード更新】
13. 「+ アクションを追加」をクリック
14. アクションの種類で「レコードを更新」を選択
15. 対象レコード: アクション1（レコード検索）の検索結果を選択
16. 更新フィールドを設定:
    - fld_clock_out（退勤時刻） = トリガーレコードの fld_clock_out（退勤時刻）
    - fld_gps_out（退勤GPS） = トリガーレコードの fld_gps_out（退勤GPS）
17. 「保存」をクリック
18. ルールのスイッチをONにして有効化
```

### テスト方法

```
【正常系テスト】
1. T4に出勤レコードを作成（テスト作業員、本日日付、打刻種別=出勤、出勤時刻=08:00）
2. T4に退勤レコードを作成（同一作業員、同一日付、同一案件、打刻種別=退勤、退勤時刻=17:00）
3. 確認: 出勤レコードの fld_clock_out が 17:00 に更新されていること
4. 確認: 出勤レコードの fld_actual_hours（実働時間）が自動計算されていること

【異常系テスト】
5. 出勤レコードが存在しない状態で退勤レコードを作成
6. 確認: エラーにならず、退勤レコードのみが残ること

【複数案件テスト】
7. 同一作業員が異なる案件に出勤している場合、正しい案件の出勤レコードに転記されること
```

### 制約事項・注意点

| 制約 | 影響 | 回避策 |
|------|------|--------|
| Bitableの検索アクションは完全一致のみ | 日付のフォーマット不一致で検索失敗の可能性 | フォーム側で日付フォーマットを統一する |
| 同一日・同一案件に複数回出勤した場合 | 最新の出勤レコードにのみ転記される | 運用ルールとして1日1案件1打刻を徹底する |
| 自動化の実行タイミング | レコード追加の数秒後に実行される（即時ではない） | 退勤打刻後、数秒待ってからレコードを確認する |
| 地理情報フィールドの転記 | Bitableのバージョンにより地理情報の転記が非対応の場合がある | 非対応の場合はテキストフィールドに緯度・経度を文字列として記録する代替案を検討 |

## A2: 日報未提出リマインダー

### 基本情報

| 項目 | 内容 |
|------|------|
| ルール名 | 日報未提出リマインダー |
| 目的 | 出勤記録があるのに日報が未提出の作業員に通知する |
| トリガー | 定時実行: 毎日 18:00 |
| 優先度 | 高 |

> 注記: 承認状態=「未提出」のレコードは、管理者が手動で作成した下書きレコードを想定しています。通常運用では、フォーム送信時に自動的に「提出済」となります。A2（日報未提出リマインダー）は、出勤記録があるのに日報レコード自体が存在しない作業員を検知する設計です。

### アクション詳細

| ステップ | アクション | 対象 |
|---------|-----------|------|
| 1 | レコード検索（出勤者） | T4を検索: 打刻日=今日 AND 打刻種別=出勤 |
| 2 | レコード検索（日報提出者） | T5を検索: 報告日=今日 AND 承認状態 != 未提出 |
| 3 | 差分抽出 | T4に存在しT5に存在しない作業員を特定 |
| 4 | 通知送信 | 該当作業員にLark通知を送信 |

### 通知内容

```
本日の日報が未提出です。18:30までに提出してください。
日報フォーム: [リンク]
```

### Bitableの定時トリガーの制約と代替案

Bitableの自動化機能では、定時トリガー（スケジュール実行）は「テーブル内の条件に一致するレコードに対してアクションを実行する」形式であり、2つのテーブルを跨いだ差分チェック（出勤はあるが日報がない作業員を特定する）を単一の自動化ルールで完全に実現することは困難である。

#### 推奨案: 簡易版（全員リマインド方式）

T4の本日の出勤レコードを検索し、該当する全作業員にリマインダーを送信する方式。日報提出済みの作業員にも通知が届くが、実装がシンプルで確実に動作する。

#### 代替案: フラグ方式（差分チェック対応）

T4に「日報提出済」チェックボックスフィールド（`fld_report_submitted`）を追加する。
- A2-a: T5にレコードが追加された時、対応するT4のレコードの`fld_report_submitted`をONにする
- A2-b: 定時実行（18:00）で、T4から「打刻日=今日 AND 打刻種別=出勤 AND fld_report_submitted=OFF」のレコードを検索し、該当作業員に通知

### 設定手順（Bitable UIでの操作ステップ）

以下は推奨案（簡易版）の手順。

```
1. Bitableを開き、「自動化」をクリック
2. 「+ ルールを作成」をクリック
3. ルール名に「A2: 日報未提出リマインダー」と入力

【トリガー設定】
4. トリガーの種類で「定時実行」を選択
5. 実行頻度: 「毎日」を選択
6. 実行時刻: 「18:00」を設定

【アクション1: レコード検索】
7. 「+ アクションを追加」をクリック
8. アクションの種類で「レコードを検索」を選択
9. 検索先テーブル: 「T4: 出退勤記録（tbl_attendance）」
10. 検索条件:
    - fld_att_date（打刻日） = TODAY()（本日の日付）
    - fld_punch_type（打刻種別） = 「出勤」

【アクション2: 通知送信】
11. 「+ アクションを追加」をクリック
12. アクションの種類で「通知を送信」を選択
13. 通知先: 検索結果の fld_att_worker（作業員）→ ルックアップで fld_lark_account を参照
14. 通知内容:
    「本日の日報が未提出です。18:30までに提出してください。」
15. 「保存」をクリック
16. ルールのスイッチをONにして有効化
```

以下はフラグ方式（差分チェック対応）を実装する場合の追加手順。

```
【事前準備: T4にフィールド追加】
0-a. T4に「日報提出済」チェックボックスフィールド（fld_report_submitted）を追加
0-b. デフォルト値: OFF

【補助ルール A2-a: 日報提出時にフラグ更新】
1. 自動化ルールを新規作成、名前: 「A2-a: 日報提出フラグ更新」
2. トリガー: T5にレコードが追加された時
3. アクション1: T4を検索
   - fld_att_worker = トリガーレコードの fld_reporter
   - fld_att_date = トリガーレコードの fld_report_date
   - fld_punch_type = 「出勤」
4. アクション2: レコード更新
   - 検索結果の fld_report_submitted = ON

【メインルール A2-b: 定時リマインダー】
5. トリガー: 定時実行、毎日18:00
6. アクション1: T4を検索
   - fld_att_date = TODAY()
   - fld_punch_type = 「出勤」
   - fld_report_submitted = OFF
7. アクション2: 検索結果の作業員に通知送信
```

### テスト方法

```
【簡易版テスト】
1. T4に本日の出勤レコードを作成
2. 自動化ルールを手動実行（テスト実行ボタン）
3. 確認: 出勤レコードの作業員にLark通知が届くこと

【フラグ方式テスト】
1. T4に本日の出勤レコードを作成（fld_report_submitted = OFF）
2. T5に該当作業員の日報を作成
3. 確認: T4の fld_report_submitted が ON に更新されること
4. 自動化ルールA2-bを手動実行
5. 確認: fld_report_submitted = OFF のレコードの作業員のみに通知が届くこと
```

### 制約事項・注意点

| 制約 | 影響 | 回避策 |
|------|------|--------|
| 2テーブル間の差分チェックが単一ルールでは困難 | 未提出者のみへの通知が直接的にはできない | フラグ方式で補助ルールを組み合わせる |
| 定時トリガーの実行時刻は分単位の指定不可（一部プラン） | 厳密に18:00丁度ではなくずれる場合がある | 数分の誤差を許容する運用とする |
| 通知先のLarkアカウント解決 | リンクフィールド経由の通知先指定が複雑 | ルックアップフィールドでLarkアカウントをT4に引き込んでおく |

## A3: 承認依頼通知

### 基本情報

| 項目 | 内容 |
|------|------|
| ルール名 | 承認依頼通知 |
| 目的 | 日報が提出されたことを班長に通知し、承認作業を促す |
| トリガー | T5（日報）のフィールド値が変更された時 |
| トリガー条件 | `fld_report_approval`（承認状態）が「提出済」に変更 |
| 優先度 | 高 |

### アクション詳細

| ステップ | アクション | 対象 |
|---------|-----------|------|
| 1 | 通知先の特定 | T2（作業員マスタ）から `fld_is_leader` = true の作業員を検索 |
| 2 | 通知送信 | 該当班長のLarkアカウントに通知を送信 |

### 通知内容

```
{報告者}さんの日報が提出されました。確認・承認をお願いします。
対象日: {報告日}
案件: {対象案件}
レコードリンク: [日報を確認する]
```

### 設定手順（Bitable UIでの操作ステップ）

```
1. Bitableを開き、「自動化」をクリック
2. 「+ ルールを作成」をクリック
3. ルール名に「A3: 承認依頼通知」と入力

【トリガー設定】
4. トリガーの種類で「レコードが更新された時」を選択
5. テーブルで「T5: 日報（tbl_daily_reports）」を選択
6. 監視フィールド: fld_report_approval（承認状態）を選択
7. 「条件を追加」をクリック
8. 条件: fld_report_approval（承認状態） = 「提出済」

【アクション1: 班長を検索】
9. 「+ アクションを追加」をクリック
10. アクションの種類で「レコードを検索」を選択
11. 検索先テーブル: 「T2: 作業員マスタ（tbl_workers）」
12. 検索条件:
    - fld_is_leader（班長フラグ） = true
    - fld_worker_status（ステータス） = 「稼働中」
    ※ 同じ班の班長に限定したい場合は、報告者の所属班と一致する条件を追加

【アクション2: 通知送信】
13. 「+ アクションを追加」をクリック
14. アクションの種類で「通知を送信」を選択
15. 通知先: アクション1の検索結果の fld_lark_account（Larkアカウント）
16. 通知内容:
    「{fld_reporter}さんの日報が提出されました。確認・承認をお願いします。
    対象日: {fld_report_date}
    案件: {fld_report_project}」
17. レコードリンクを含めるオプションがあれば有効にする
18. 「保存」をクリック
19. ルールのスイッチをONにして有効化
```

### テスト方法

```
1. T5に日報レコードを作成（承認状態 = 未提出）
2. 承認状態を「提出済」に変更
3. 確認: 班長フラグがONの作業員にLark通知が届くこと
4. 確認: 通知内容に報告者名、報告日、対象案件が含まれていること
5. 確認: 通知のレコードリンクをクリックすると該当日報レコードが開くこと
```

### 制約事項・注意点

| 制約 | 影響 | 回避策 |
|------|------|--------|
| 班の概念がBitableのリレーションで直接表現されていない | 全ての班長に通知が届く可能性がある | T2に「所属班」フィールドがあるため、報告者の所属班と一致する班長のみを検索条件に含める |
| レコードリンクの自動挿入 | Bitableのバージョンによりリンク自動挿入が非対応 | 通知本文にBase URLとレコードIDを手動で組み立てるか、非対応の場合は省略する |
| フォーム送信時のステータス変更タイミング | フォーム送信で「提出済」に変わるタイミングと自動化のトリガー検知にラグがある | フォームの送信後処理で承認状態を「提出済」に設定する運用を徹底する |

## A4: 承認フロー進行

### 基本情報

| 項目 | 内容 |
|------|------|
| ルール名 | 承認フロー進行 |
| 目的 | 班長承認が完了した日報を現場監督に通知し、最終承認を促す |
| トリガー | T5（日報）のフィールド値が変更された時 |
| トリガー条件 | `fld_report_approval`（承認状態）が「班長承認」に変更 |
| 優先度 | 高 |

### アクション詳細

| ステップ | アクション | 対象 |
|---------|-----------|------|
| 1 | 現場監督の特定 | T1（案件マスタ）から対象案件の`fld_site_manager`（現場監督）を取得 |
| 2 | 通知送信 | 現場監督のLarkアカウントに通知を送信 |

### 通知内容

```
{報告者}さんの日報が班長承認されました。最終承認をお願いします。
対象日: {報告日}
案件: {対象案件}
レコードリンク: [日報を確認する]
```

### 設定手順（Bitable UIでの操作ステップ）

```
1. Bitableを開き、「自動化」をクリック
2. 「+ ルールを作成」をクリック
3. ルール名に「A4: 承認フロー進行」と入力

【トリガー設定】
4. トリガーの種類で「レコードが更新された時」を選択
5. テーブルで「T5: 日報（tbl_daily_reports）」を選択
6. 監視フィールド: fld_report_approval（承認状態）を選択
7. 「条件を追加」をクリック
8. 条件: fld_report_approval（承認状態） = 「班長承認」

【アクション1: 案件情報の取得】
9. 「+ アクションを追加」をクリック
10. アクションの種類で「レコードを検索」を選択
11. 検索先テーブル: 「T1: 案件マスタ（tbl_projects）」
12. 検索条件:
    - レコード = トリガーレコードの fld_report_project（対象案件）のリンク先
    ※ リンクフィールド経由で直接参照できる場合は検索ステップを省略可

【アクション2: 通知送信】
13. 「+ アクションを追加」をクリック

## A7: GPS異常検知

### 基本情報

| 項目 | 内容 |
|------|------|
| ルール名 | GPS異常検知 |
| 目的 | 出勤打刻位置が現場から離れている場合に管理者に通知する |
| トリガー | T4（出退勤記録）にレコードが追加された時 |
| トリガー条件 | `fld_punch_type`（打刻種別）= 「出勤」 |
| アラート条件 | 出勤GPSと案件マスタのGPS座標の距離 > GPS許容半径 |
| 優先度 | 中 |

### 距離計算の考え方

Bitableの数式フィールドではHaversine公式（球面三角法）の完全な実装は困難であるため、日本の緯度帯（北緯30度〜45度）で有効な簡易近似式を使用する。

#### 簡易距離計算式

```
距離(m) = SQRT((lat1 - lat2)^2 + (lng1 - lng2)^2) * 111000
```

- `lat1`, `lng1`: 出勤GPS座標（緯度・経度）
- `lat2`, `lng2`: 案件マスタのGPS座標（`fld_site_lat`, `fld_site_lng`）
- `111000`: 緯度1度あたりの距離（約111km）をメートルに換算
- 日本の緯度帯では経度方向の補正係数（cos(lat) ≒ 0.77〜0.87）が必要だが、許容半径が数百メートルの判定では実用上十分な精度

#### より正確な近似式（推奨）

```
距離(m) = SQRT(
  ((lat1 - lat2) * 111000)^2 +
  ((lng1 - lng2) * 111000 * COS(RADIANS((lat1 + lat2) / 2)))^2
)
```

### 実装方式の選択肢

Bitableの自動化機能では地理情報フィールドの座標値を直接数値として取得・比較することが困難な場合がある。以下に実装方式の選択肢を示す。

#### 方式A: 数式フィールドによる事前計算（推奨）

T4に以下のフィールドを追加し、レコード作成時に自動計算させる。

| フィールド名 | フィールドID | 型 | 計算内容 |
|-------------|------------|-----|---------|
| 打刻緯度 | `fld_punch_lat` | 数値 | 出勤GPS（地理情報）から手動入力、またはフォームで別途取得 |
| 打刻経度 | `fld_punch_lng` | 数値 | 同上 |
| 現場距離(m) | `fld_site_distance` | 数式 | 上記の簡易距離計算式 |
| GPS異常フラグ | `fld_gps_alert` | 数式 | `IF(fld_site_distance > ルックアップ(fld_gps_radius), true, false)` |

自動化ルールのトリガー条件: `fld_gps_alert` = true

#### 方式B: 外部Webhookによる計算（高精度が必要な場合）

1. T4にレコードが追加された時、WebhookでGPS座標を外部サービスに送信
2. 外部サービスでHaversine公式による正確な距離計算を実行
3. 閾値超過の場合はLark APIで通知を送信

### 設定手順（Bitable UIでの操作ステップ）

以下は方式A（数式フィールドによる事前計算）の手順。

```
【事前準備: T4にフィールド追加】
0-a. T4に以下のフィールドを追加:
    - fld_punch_lat（打刻緯度）: 数値型、小数点以下6桁
    - fld_punch_lng（打刻経度）: 数値型、小数点以下6桁
    ※ 出勤フォームで緯度・経度を別フィールドとしても取得するか、
      地理情報フィールドから座標値を抽出するルックアップを設定

    - fld_lookup_site_lat: T1の fld_site_lat を参照

【トリガー設定】
4. トリガーの種類で「レコードが追加された時」を選択
5. テーブルで「T4: 出退勤記録（tbl_attendance）」を選択
6. 「条件を追加」をクリック
7. 条件:
    - fld_punch_type（打刻種別） = 「出勤」
    - fld_gps_alert（GPS異常フラグ） = true

【アクション1: 現場監督を検索】
8. 「+ アクションを追加」をクリック
9. アクションの種類で「レコードを検索」を選択
10. 検索先テーブル: 「T1: 案件マスタ（tbl_projects）」
11. 検索条件:
    - レコード = トリガーレコードの fld_att_project（対象案件）のリンク先

【アクション2: 管理者に通知送信】
12. 「+ アクションを追加」をクリック
13. アクションの種類で「通知を送信」を選択
14. 通知先: アクション1の検索結果の fld_site_manager（現場監督）
15. 通知内容:
    「[警告] {fld_att_worker}さんの出勤打刻位置が現場から離れています。確認してください。
    日付: {fld_att_date}
    案件: {fld_att_project}
    現場からの距離: 約{fld_site_distance}m
    許容半径: {fld_lookup_gps_radius}m」

### テスト方法

```

   → 約1.6kmの距離 → 閾値超過
3. 確認: GPS異常フラグがtrueになること
4. 確認: 現場監督にLark通知が届くこと

5. T4に出勤レコードを作成（打刻緯度=35.6815, 打刻経度=139.7675）
   → 約50mの距離 → 閾値以内
6. 確認: GPS異常フラグがfalseになること
7. 確認: 通知が送信されないこと

【GPS未入力テスト】
8. GPS座標が入力されていない出勤レコードを作成

```

### 制約事項・注意点

| 制約 | 影響 | 回避策 |
|------|------|--------|
| 地理情報フィールドから緯度・経度を数値として取得できない場合がある | 距離計算ができない | フォーム側で緯度・経度を数値フィールドとして別途取得する。JavaScriptウィジェットやLark Flowで座標抽出を行う方法も検討 |
| 簡易距離計算式の精度 | 経度方向の誤差が大きくなる（特に高緯度地域） | 推奨の近似式（cos補正あり）を使用する。それでも精度が不足する場合は方式Bを検討 |
| 案件マスタにGPS座標が未設定の場合 | 距離計算ができない | GPS座標未設定の案件ではアラートをスキップする条件を追加する |
| 屋内・地下での GPS 精度 | GPS信号が弱い環境では座標が大きくずれる | 許容半径を広めに設定する（例: 1000m）。GPS精度が低い旨を作業員に周知する |
| 直行・直帰の場合 | 自宅等から直行する場合、現場以外からの打刻が正当なケースがある | T4の `fld_att_status`（出勤ステータス）が「直行」の場合はアラートを除外する条件を追加する |

## Bitableの自動化制限事項

Lark Bitableの自動化機能には以下の制限がある。設計時にこれらを把握し、適切な回避策を講じること。

### 1. ルール数・実行回数の制限

| 項目 | 制限値 | 備考 |
|------|--------|------|
| 1ベースあたりの自動化ルール数 | 最大50ルール（プランにより異なる） | 本設計では7ルール（+補助ルール2）で合計9ルール |
| 月間実行回数 | プランにより上限あり（Standardプランで5,000回/月程度） | 作業員数 x 営業日数で見積もること |
| 1回の実行で処理できるレコード数 | 検索結果は最大500件 | 大量データの一括処理には不向き |

### 2. トリガー種類の制約

| トリガー種類 | 利用可否 | 備考 |
|-------------|---------|------|
| レコードが追加された時 | 可 | A1, A7で使用 |
| レコードが更新された時 | 可 | A3, A4, A5, A6で使用。監視フィールドの指定が必要 |
| 定時実行（スケジュール） | 可 | A2で使用。日次・週次・月次の指定が可能 |
| フォーム送信時 | 可（一部プラン） | トリガーとして利用可能だが、レコード追加トリガーで代替可 |
| Webhook受信 | 可（Enterpriseプラン） | 外部システム連携に利用 |
| レコード削除時 | 不可の場合あり | 削除イベントのトリガーは非対応のバージョンがある |

### 3. 条件・アクションの制約

| 制約 | 詳細 | 回避策 |
|------|------|--------|
| 複雑な条件分岐（IF/ELSE） | 自動化内でのIF分岐は限定的 | 条件ごとに別ルールを作成する |
| 数式フィールドの値を条件に使用 | 一部のバージョンで非対応 | フラグ用チェックボックスを追加して対応 |
| 複数テーブルの結合検索 | 単一テーブルの検索のみ | ルックアップフィールドで事前にデータを引き込む |
| ループ処理 | 非対応 | 検索結果の全レコードに対して一括アクションを適用する形式のみ |
| 外部API呼び出し | Enterpriseプランで「HTTPリクエスト」アクションが利用可能 | 無い場合はLark Flowとの組み合わせで対応 |
| 変数の保持 | 自動化実行中の中間値を変数として保持できない | 検索結果を次のアクションの入力として渡す形式で代替 |

### 4. 通知に関する制約

| 制約 | 詳細 | 回避策 |
|------|------|--------|
| 通知先の動的指定 | ユーザー型フィールドまたは検索結果からの指定が必要 | ルックアップフィールドでLarkアカウントを引き込んでおく |
| 通知テンプレートの書式 | リッチテキスト（太字・リンク等）の制限がある | 簡潔なプレーンテキストで記述する |
| レコードリンクの自動挿入 | 一部のバージョンで非対応 | 通知に手動でURLを組み立てるか、省略する |
| 通知頻度の制限 | 短時間に大量の通知を送ると制限がかかる場合がある | バッチ通知（まとめて通知）を検討する |

### 5. 回避策と代替手段

Bitableの自動化だけでは実現が困難な処理については、以下の代替手段を検討する。

| 代替手段 | 適用場面 | 難易度 |
|---------|---------|--------|
| **Lark Flow（ワークフロー）** | 複雑な条件分岐、複数テーブルの結合処理、外部API連携 | 中 |
| **Lark Bot（カスタムボット）** | 高度な通知制御、インタラクティブな承認フロー | 高 |
| **Webhook + 外部サーバー** | 距離計算、データ分析、外部システム連携 | 高 |
| **定期手動確認** | 複雑な差分チェック、例外処理 | 低 |

## 自動化ルールの導入順序

運用開始時は全ルールを一度に有効化せず、段階的に導入することを推奨する。

| フェーズ | 対象ルール | 目的 | 期間目安 |
|---------|-----------|------|---------|
| Phase 1 | A1（退勤データ統合） | データ統合の基盤を確立 | 1週目 |
| Phase 2 | A3, A4, A5（承認フロー系） | 承認プロセスの自動化 | 2週目 |
| Phase 3 | A2（日報リマインダー） | 入力漏れの防止 | 3週目 |
| Phase 4 | A6, A7（アラート系） | 安全管理の強化 | 4週目 |

各フェーズで1週間程度の運用テストを行い、問題がないことを確認してから次のフェーズに進む。

## 運用・保守

### 自動化ルールの監視

- Bitableの自動化実行ログを定期的に確認し、エラーが発生していないかチェックする
- 月間実行回数がプラン上限に近づいていないか監視する
- 通知が正しく届いているか、受信者に定期的にヒアリングする

### ルール変更時の注意

- ルールを変更する前に、現在のルール設定のスクリーンショットを取得しておく
- 変更後はテスト実行で動作確認を行い、本番データに影響がないことを確認する
- 複数ルールが連動している場合（A1 → A6 など）、一方の変更が他方に影響しないか確認する
