# ダッシュボード設計書（Dashboard Design）

建設業向け勤怠・日報テンプレート — Lark Bitable ダッシュボード定義

---

## ダッシュボード一覧

| # | ダッシュボード名 | 対象ユーザー | 目的 | 更新頻度 |
|---|-----------------|-------------|------|---------|
| D1 | 現場日次ダッシュボード | 現場監督・班長 | 今日の現場状況をリアルタイムで把握 | リアルタイム |
| D2 | 月次管理ダッシュボード | 管理者・経営層 | 月間の勤怠・工数・コストを把握 | 日次更新 |

## D1: 現場日次ダッシュボード

**目的**: 今日の現場状況をリアルタイムで把握する。朝礼時の出勤確認、日中の進捗管理、夕方の日報提出督促に活用。

**対象ユーザー**: 現場監督、班長

| # | ウィジェット名 | 種類 | データソース |
|---|--------------|------|-------------|
| D1-W1 | 本日の出勤者数 | 数値カード | T4: 出退勤記録 |
| D1-W2 | 出勤状況一覧 | テーブル | T4: 出退勤記録 |
| D1-W3 | 案件別出勤者数 | 棒グラフ | T4: 出退勤記録 |
| D1-W4 | 日報提出状況 | 円グラフ | T5: 日報 |

### D1-W4: 日報提出状況

| 項目 | 設定値 |
|------|--------|
| **ウィジェット種類** | 円グラフ（Pie Chart） |
| **データソース** | `tbl_daily_reports`（T5: 日報）、`tbl_attendance`（T4: 出退勤記録） |
| **フィルタ条件** | `fld_report_date` = TODAY()（T5）/ `fld_att_date` = TODAY()（T4） |
| **集計方法** | 提出済 = COUNT(`fld_report_id`) WHERE `fld_report_approval` != "未提出"、未提出 = 本日出勤者数 - 提出済人数 |
| **グループ化** | 提出済 / 未提出 の2区分 |
| **色設定** | 提出済: 緑（#52C41A）、未提出: 赤（#FF4D4F） |
| **表示形式** | ドーナツ型、中央に提出率（%）表示 |

**補足:** Lark Bitableの標準機能では出退勤テーブルと日報テーブルのクロス集計が難しいため、以下の代替実装を推奨。
- **代替案A**: T5の `fld_report_approval` でグループ化し、「未提出」「提出済」「班長承認」「監督承認」「差戻し」の5区分で円グラフを作成。
- **代替案B**: T5のフィルタを `fld_report_date` = TODAY() とし、`fld_report_approval` の値でグループ化。「未提出」とそれ以外の割合を表示。

### D1-W5: 未承認件数

| 項目 | 設定値 |
|------|--------|
| **ウィジェット種類** | 数値カード（Stat Card） |
| **データソース** | `tbl_attendance`（T4）+ `tbl_daily_reports`（T5） |
| **フィルタ条件** | `fld_att_approval` = "未承認"（T4）/ `fld_report_approval` = "提出済"（T5、未承認だが提出済のもの） |
| **集計方法** | COUNT（T4の未承認） + COUNT（T5の提出済で未承認） |
| **表示形式** | 大きな数値 + 「件」の単位表示 |
| **色設定** | 橙系（#FA8C16）、0件の場合は緑系（#52C41A） |

**補足:** Lark Bitableでは1つの数値カードに複数テーブルの集計を合算できないため、以下の分割実装を推奨。
- **数値カードA**: 出退勤の未承認件数（T4: `fld_att_approval` = "未承認" のCOUNT）
- **数値カードB**: 日報の未承認件数（T5: `fld_report_approval` = "提出済" のCOUNT）

### D1-W6: 天候

| 項目 | 設定値 |
|------|--------|
| **ウィジェット種類** | テキスト / 数値カード（Stat Card） |
| **データソース** | `tbl_attendance`（T4: 出退勤記録） |
| **フィルタ条件** | `fld_att_date` = TODAY() AND `fld_punch_type` = "出勤" |
| **集計方法** | `fld_weather` の最頻値（MODE）、または最初の1レコードの値 |
| **表示形式** | テキスト表示（例: "晴"） |
| **色設定** | 晴: 橙（#FA8C16）、曇: 灰（#8C8C8C）、雨: 青（#1890FF）、雪: 水色（#69C0FF）、強風: 紫（#722ED1） |

**補足:** Lark Bitableのダッシュボードではテキストウィジェットへの動的バインドが制限されるため、以下の代替実装を推奨。
- **代替案**: 棒グラフで `fld_weather` をグループ化し、各天候の出現回数を表示。最も多い天候が本日の天候となる。

### D2-W7: 案件別労務費

## 権限設計

### ダッシュボード閲覧権限

| ダッシュボード | 管理者 | 現場監督 | 班長 | 職人（一般作業員） |
|--------------|--------|---------|------|------------------|
| D1: 現場日次 | 閲覧可 | 閲覧可 | 閲覧可 | 閲覧可（自身の案件のみ） |
| D2: 月次管理 | 閲覧可 | 閲覧可（担当案件のみ） | 閲覧不可 | 閲覧不可 |
| D3: 案件別詳細 | 閲覧可 | 閲覧可（担当案件のみ） | 閲覧不可 | 閲覧不可 |

### 権限制御の実装方法

Lark Bitableのダッシュボード権限は、Bitableのアクセス権限で制御する。

1. **管理者**: Bitable全体の管理者権限を付与。全ダッシュボードにアクセス可能。
2. **現場監督**: D1は全体閲覧可。D2・D3は担当案件（`fld_site_manager` = 自身）に紐づくデータのみフィルタで表示。
3. **班長**: D1のみ閲覧可。所属班のメンバーの出勤状況が確認可能。
4. **職人（一般作業員）**: D1のみ閲覧可。自身の出退勤記録と日報の状況が確認可能。

**実装上の注意:**
- Lark Bitableではダッシュボードレベルでのロールベースアクセス制御（RBAC）は限定的。
- 案件単位のデータ制限は、ダッシュボード共有リンクのフィルタ固定機能を活用する。
- 詳細な権限制御が必要な場合は、役職別に専用ダッシュボードを作成する運用を推奨。

### ダッシュボード確認タイミング

### トラブルシューティング

| 事象 | 原因 | 対処法 |
|------|------|--------|
| 数値が0と表示される | フィルタ条件が不正、またはデータ未登録 | フィルタの日付範囲を確認 |
| グラフが表示されない | データ件数が0 | 対象期間にデータが存在するか確認 |
| 労務費が計算されない | T6の集計が未実行 | 月次集計処理を手動実行 |
| 権限エラー | Bitable共有設定の不備 | 管理者がダッシュボードの共有設定を確認 |
