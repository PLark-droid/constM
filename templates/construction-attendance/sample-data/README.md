# サンプルデータ

## 概要

テンプレートのテスト・デモ用ダミーデータです。実際の人物・案件とは関係ありません。
注意: 本CSVはA1（退勤データ統合）実行後の統合済み状態です。A1のテストには、出勤打刻フォームと退勤打刻フォームから個別にテスト送信してください。

### sample-daily-reports.csv（日報）

- **期間**: 2026/02/09 - 2026/02/20（平日10日間）
- **レコード数**: 99件（1日あたり約10名分）
- **対象案件**: PJ-001（新宿）およびPJ-002（横浜）

**データのバリエーション**:

| 項目 | 内容 |
|------|------|
| 作業内容数 | 1件から3件（大半は2件、残業時に3件記入のパターンあり） |
| 安全事項 | 約40%の日報に安全事項の記入あり |
| 進捗メモ | 全件記入済み。進捗率や具体的な作業内容を記載 |
| 明日の予定 | 大半に記入あり |
| 写真 | 全件空欄（テストデータのため） |

**承認状態の分布**:

| 承認状態 | 対象期間 | 説明 |
|---------|---------|------|
| 監督承認 | 02/09 | 現場監督による承認済み |
| 班長承認 | 02/10 - 02/13 | 班長による承認済み |
| 差戻し | 02/13, 02/17 | DR-0045（作業時間整合性）、DR-0066（遅刻理由記載不足） |
| 提出済 | 02/16 - 02/17 | 提出済み・未承認 |
| 未提出 | 02/18 - 02/20 | 未提出（ワークフローテスト用） |

承認状態=「未提出」のレコードは、管理者が手動で作成した下書きレコードを想定しています。通常運用では、フォーム送信時に自動的に「提出済」となります。A2（日報未提出リマインダー）は、出勤記録があるのに日報レコード自体が存在しない作業員を検知する設計です。

**カラム構成**: 日報ID, 報告者, 対象案件, 報告日, 作業内容1, 作業時間1(h), 作業内容2, 作業時間2(h), 作業内容3, 作業時間3(h), 進捗メモ, 安全事項, 明日の予定, 写真, 承認状態, 承認コメント

## 使い方

### インポート手順

1. CSVファイルをダウンロード
2. Bitableのテーブルにインポート
3. 詳細は [setup-guide/07-initial-data-setup.md](../setup-guide/07-initial-data-setup.md) 参照

### 自社データへのカスタマイズ

1. CSVをExcelで開く
2. サンプルデータを自社の情報に書き換え
3. UTF-8でCSV保存
4. Bitableにインポート

## 注意事項

- **エンコーディング**: UTF-8（BOM付き推奨）
- **日付形式**: YYYY/MM/DD
- **GPS座標**: 小数点以下6桁
- **リンクフィールド**（案件名、作業員名等）: インポート後にBitable上でリンク設定が必要な場合あり

## データ間の整合性

サンプルデータは以下の整合性ルールに従って作成されています。

- 出退勤データの「作業員」列と「対象案件」列の値は、master-workers.csv および master-projects.csv の値と完全一致
- 日報データの作業内容名（作業内容1/2/3列）は、master-work-types.csv の「作業名」列と完全一致
- 出退勤と日報は同一日・同一作業員で対応関係がある（打刻日 = 報告日、作業員 = 報告者）
- C班（松本龍二・井上拓也）はPJ-003所属だが、PJ-003が準備中のため出退勤・日報ではPJ-001（新宿現場）に応援出勤している
